# Пайплайн До 4 Этапа: Только Решающие Специалисты

Статус: архивный подробный черновик до стабилизации нейминга.

```text
не использовать как активный маршрут
активный маршрут сейчас:
02_routing/READY_PIPELINE_01_05_RU.md
02_routing/READY_PIPELINE_06_07_RU.md
00_navigation/CANONICAL_NAMING_AND_FOLDER_MAP.md
```

Статус: упрощённая схема. Без обслуживающих ролей.

Важно:

```text
актуальный source of truth по маршруту и ролям сейчас:
02_routing/READY_PIPELINE_01_05_RU.md
03_roles/ACTIVE_TEAM_GROUPS_AND_ROUTING_RU.md
```

Если в этом подробном файле где-то остались старые расширенные формулировки,
ориентируемся на активные документы выше.

Главное правило:

```text
В схеме остаются только специалисты, которые принимают смысловые / стратегические решения.
AI-разметчик, аналитик отзывов, руководитель исследования и отдельный контролёр доказательности здесь не считаются специалистами.
```

---

## Специалисты, Которые Остаются

### 1. Маркетплейс-стратег / Категорийный стратег

Решает:

- что покупатель увидит в отзывах как риск;
- что является нормой категории;
- что является паритетом;
- что может быть преимуществом героя;
- какие сигналы конкурентов важны для сравнения.

### 2. Поведенческий стратег

Решает:

- для кого минус станет стоп-фактором;
- для кого минус терпим;
- почему покупатель всё равно может хотеть продукт;
- какие позитивы защищают желание купить;
- какие сценарии меняют восприятие минуса.

### 3. Специалист по продуктовой честности и обещаниям

Решает:

- что нельзя превращать в обещание;
- где будущие обещания могут спорить с отзывами;
- какие формулировки опасны;
- какие идеи физически / честно не держатся продуктом.

### 4. Стратег позиционирования / Креативный стратег

До 4 этапа не выбирает финальное позиционирование.

Но может участвовать на 4 этапе как стратегический редактор:

- какие строки матрицы потенциально могут стать ставкой;
- какие строки оставить как support;
- какие строки только банк идей;
- какие строки не стоит тащить дальше.

## Этап 1. Intake / Сбор Сырья

### Задача этапа

Собрать и разложить материалы:

- герой;
- прямые конкуренты;
- косвенные конкуренты;
- отзывы;
- карточки как источник фактов: описание, характеристики, вес / объём;
- продуктовый контекст;
- choice context: цена, цены прямых конкурентов, объём / цена за единицу, рейтинг + количество отзывов.

### Кто принимает решения

Специалисты не подключаются.

Это техническая раскладка сырья.

### Что решается

- кто герой;
- кто прямой конкурент;
- кто косвенный конкурент;
- какие отзывы нужно разметить;
- каких материалов не хватает;
- хватает ли якорного контекста, чтобы потом оценивать терпимость к минусам.

### Выход

`01__project__intake_map.md`

В нём:

```text
hero:
direct_competitors:
indirect_competitors:
review_sources:
card_sources:
missing_materials:
```

Если нет цены, объёма / цены за единицу, рейтинга или описания / характеристик товара, это фиксируется сразу.

Без этого нельзя уверенно оценивать терпимость к минусу.

## Этап 1.5. Product Facts And Price Snapshot / Факты Продукта И Снимок Цен

### Задача этапа

Быстро зафиксировать только факты, без интерпретации карточки:

- описание селлера;
- обещания селлера;
- характеристики;
- вес / объём / количество;
- цены;
- объём / цену за единицу;
- рейтинг и количество отзывов.

Это нужно не для построения воронки.

Это нужно только чтобы:

- корректно считать цену за единицу;
- понимать, что физически продаётся;
- видеть заявленные селлером claims;
- не оценивать терпимость к минусу без цены.

### Кто принимает решения

Финальных стратегических решений здесь ещё нет.

Это подготовительная ветка:

- пользователь даёт / прикладывает факты карточки;
- AI структурирует факты;
- специалисты используют это позже на этапах 3 и 4.

### Выходы

```text
01_intake/product_passports/{role}/{product_key}/01__product_key__product_passport.md
01_intake/01__project__price_value_snapshot.md
```

Паспорта раскладываются сразу по роли:

```text
hero
direct_competitors
indirect_competitors
```

Если роль временно непонятна:

```text
_unclassified
```

Но до этапа 3 продукт должен быть отнесён к прямым или косвенным.

### Где Фиксируются Цены Конкурентов

Цены конкурентов фиксируются в:

```text
01_intake/01__project__price_value_snapshot.md
```

Туда входят:

- герой;
- все прямые конкуренты;
- косвенные конкуренты только если они реально участвуют в выборе как альтернатива.

### Что НЕ делаем

- не анализируем выдачу;
- не критикуем карточки;
- не фиксируем сильные / слабые визуальные ходы;
- не строим пропозицию;
- не делаем первое впечатление покупателя;
- не парсим рынок, если техническая цена высокая;
- не делаем финальную стратегию.

### Правило Скорости

Основной вход:

```text
скриншоты + скопированный текст описания / характеристик + 5 коротких полей: цена, объём, рейтинг, число отзывов, ссылка + голосовые claims с картинок
```

Если можно быстро скопировать страницу, копируем.

Если нужно долго вручную собирать паспорт, не делаем.

ИИ создаёт:

```text
01_intake/product_passports/{role}/{product_key}/01__product_key__product_passport.md
```

и забирает цену / объём / рейтинг в:

```text
01__project__price_value_snapshot.md
```

Подробный метод:

```text
04_methods/PRODUCT_FACTS_PRICE_AND_DEFERRED_CARD_CRITIQUE_METHOD.md
```

Критика карточек и выдачи не используется до 4 этапа.

На этапе 4 она открывается как человеческий слой восприятия рынка:

```text
04 Matrix Conclusions
05 Strategy Decision
06 Funnel Architecture
07 Card Layer
```

## Этап 2. Full Signal Markup / Полная Разметка Всех Отзывов

### Задача этапа

Полностью разметить отзывы героя, прямых конкурентов и косвенных конкурентов.

### Кто принимает решения

Специалисты не подключаются.

Это технический AI-прогон.

### Что решается

Стратегически ничего не решается.

AI только раскладывает отзывы на сигналы:

- позитив;
- негатив;
- ожидание;
- сомнение;
- сравнение;
- сценарий;
- manual queue;
- trash.

### Что важно

На этом этапе нельзя:

- выбирать углы;
- решать, какие минусы важнее;
- строить сделки;
- сравнивать конкурентов;
- искать "правду о продукте".

### Выход

```text
02__hero__signal_markup.md
02__direct_competitor_key__signal_markup.md
02__indirect_competitor_key__signal_markup.md
```

## Этап 3. Матрицы Контрактов

Чтобы не плодить лишние слои, для человека это теперь один этап.

Внутри него есть подшаги, но на человеческом уровне логика одна:

```text
сначала собираем локальные матрицы по продуктам
потом обновляем одну мастер-матрицу героя
```

Это значит:

- герой получает свою локальную матрицу;
- каждый прямой конкурент получает свою локальную матрицу в своей папке;
- каждый косвенный конкурент получает свою локальную матрицу в своей папке;
- итоговый рабочий стол этапа — мастер-матрица героя.

### Какие вопросы должен решить готовый этап 3

1. Какие контракты героя вообще реально видны в отзывах?
2. Какие минусы для каких людей являются стопом?
3. Какие минусы терпимы и при каком условии?
4. Что удерживает желание купить, несмотря на минус?
5. Какие claims селлера отзывы держат, а какие делают опасными?
6. Какие прямые конкуренты подтверждают норму категории?
7. Какие косвенные конкуренты открывают важный соседний сценарий?
8. Какие черновые контракты героя после конкурентов подтверждаются?
9. Какие строки надо ослабить или выкинуть?
10. Что реально передаётся дальше в стратегию?

### Кто принимает решения

1. `Маркетплейс-стратег / Категорийный стратег`
2. `Поведенческий стратег`
3. `Специалист по продуктовой честности и обещаниям`

### Минимальная форма строки

```text
контракт_или_сценарий:
видимый_минус_или_риск:
кому_стоп:
когда_терпимо:
что_удерживает_покупку:
цена_или_контекст:
риск_обещания:
evidence_ids:
статус:
```

Статусы на старте:

```text
draft_from_hero
confirmed_by_direct
enriched_by_indirect
weakened_by_market
reject
hold
```

### Этап 3.1. Локальная Матрица Героя

Входы:

```text
02_signal_markup/hero/{product_key}/02__{product_key}__signal_markup.md
01_intake/product_passports/hero/{product_key}/01__{product_key}__product_passport.md
01_intake/01__project__price_value_snapshot.md
```

Выход:

```text
03_contract_matrices/hero/{product_key}/03__{product_key}__contracts_matrix.md
```

Здесь рождаются черновые контракты героя.

### Этап 3.2. Локальные Матрицы Прямых Конкурентов

Входы:

```text
02_signal_markup/direct_competitors/{product_key}/02__{product_key}__signal_markup.md
01_intake/product_passports/direct_competitors/{product_key}/01__{product_key}__product_passport.md
01_intake/01__project__price_value_snapshot.md
```

Выход:

```text
03_contract_matrices/direct_competitors/{product_key}/03__{product_key}__contracts_matrix.md
```

Здесь мы не строим стратегию героя.

Мы только видим:

- что является нормой категории;
- что усиливает или ослабляет контракт героя;
- что не стоит тащить дальше.

### Этап 3.3. Локальные Матрицы Косвенных Конкурентов

Входы:

```text
02_signal_markup/indirect_competitors/{product_key}/02__{product_key}__signal_markup.md
01_intake/product_passports/indirect_competitors/{product_key}/01__{product_key}__product_passport.md
01_intake/01__project__price_value_snapshot.md
```

Выход:

```text
03_contract_matrices/indirect_competitors/{product_key}/03__{product_key}__contracts_matrix.md
```

Косвенные конкуренты здесь нужны не как доказательство героя, а как:

- соседний сценарий;
- язык;
- альтернатива по удобству;
- дополнительная проверка, где минус героя становится ещё опаснее.

### Этап 3.4. Мастер-Матрица Героя

Входы:

```text
03_contract_matrices/hero/{hero_product_key}/03__{hero_product_key}__contracts_matrix.md
03_contract_matrices/direct_competitors/{product_key}/03__{product_key}__contracts_matrix.md
03_contract_matrices/indirect_competitors/{product_key}/03__{product_key}__contracts_matrix.md
```

Главный выход этапа:

```text
03_contract_matrices/hero/{hero_product_key}/03__{hero_product_key}__master_contracts_matrix.md
```

Что происходит:

- подтверждаем строки героя;
- ослабляем слабые строки;
- добираем новые только если рынок реально их открыл;
- выкидываем мусор;
- не тащим дальше всё подряд.

### Что НЕ входит в этап 3

- критика карточки;
- критика выдачи;
- первое впечатление;
- визуальные нормы;
- цветовые ассоциации;
- настроение человека;
- советы друзей;
- алгоритм до карточки.

Это не входит в этап 3.

После заморозки мастер-матрицы это можно открыть на этапе 4 как человеческий слой восприятия рынка.

## Этап 4. Выводы По Мастер-Матрице

После этапа 3 у нас уже есть очищенная база.

Теперь делаем не финальную стратегию, а один короткий выводной документ:

```text
04_matrix_conclusions/hero/{hero_product_key}/04__{hero_product_key}__matrix_conclusions.md
```

Внутри два слоя:

### 4.1. Фактические выводы по матрице

Сюда попадает только то, что уже реально держится:

- подтверждённые контракты;
- стоп-факторы;
- терпимые минусы;
- рискованные claims;
- важные рыночные нормы.

### 4.2. Интерпретационные гипотезы

Сюда попадает не факт, а аккуратная гипотеза:

- как человек может это прочитать;
- где один и тот же минус может читаться мягче;
- где риск может стать тяжелее на уровне восприятия;
- какие объяснения выбора вообще правдоподобны.

Главное правило:

```text
фактические выводы и гипотезы восприятия идут рядом,
но не смешиваются в одну кашу
```

### Кто работает

На этом этапе не делаем комитет.

Есть один владелец и несколько узких функций.

#### `owner`

`Стратег позиционирования / brand strategist`

Он собирает итоговый файл:

- блок фактических выводов;
- блок интерпретационных гипотез;
- короткую передачу дальше.

#### `input`

`Поведенческий стратег`

Даёт гипотезы чтения:

- как человек может это интерпретировать;
- где минус читается мягче или тяжелее;
- какие объяснения выбора выглядят правдоподобно.

#### `filter`

`Маркетплейс-стратег / Категорийный стратег`

Может только:

- вырезать то, что не выдерживает реальность категории;
- пометить то, что не влияет на реальный выбор в маркетплейсе.

#### `filter`

`Специалист по продуктовой честности и обещаниям`

Может только:

- вырезать опасные интерпретации;
- пометить конфликт с отзывами и claims.

#### `variant` по необходимости

`Креативный директор / creative strategist`

Не соавторит документ по умолчанию.

Подключается только если:

- owner видит реально две сильные трактовки;
- нужен альтернативный сильный рефрейм.

Тогда он даёт не компромисс, а:

```text
variant B
```

Не передаётся:

- всё, что в `hard_reject`;
- всё, что зависло в `hold` без шанса быстро закрыть дыру;
- красивые, но бесполезные наблюдения.

Материал с маршрутом `claim_to_reframe`, `fear_material`, `visible_delta`, `composition_material`, `keep_for_marketing` или `keep_for_creative` не считается мусором сам по себе.

## Этап 5. Разработка Стратегии

Теперь начинается уже не диагностика, а выбор направления.

Но этот выбор идёт не из любой теоретической стратегии, а через фильтр:

```text
авторская матрица стратегий
```

Она ограничивает фантазию клиента и возвращает проект к тому, с чем автор реально работает.

Сейчас допустимые направления такие:

- характер бренда;
- выделиться;
- ясность;
- доверие;
- премиализация.

Направление типа:

```text
перекричать конкурента
```

может существовать в мире, но по умолчанию не выбирается, потому что не согласовано со стилем проекта.

### Задача этапа

Из выводов по мастер-матрице выбрать:

- `1 основное направление`
- `1 поддерживающее направление`

И не дать им конфликтовать между собой.

### Кто принимает решения

И здесь тоже нет коллективного компромисса.

#### `owner`

`Стратег позиционирования / brand strategist`

Он выбирает:

- какие линии допустимы;
- что становится `primary`;
- что становится `support`;
- какие линии rejected.

#### `filter`

`Маркетплейс-стратег / Категорийный стратег`

Может:

- отрезать то, что плохо читается в категории;
- отрезать то, что не выдерживает marketplace reality.

#### `input`

`Поведенческий стратег`

Даёт:

- проверку человеческой правдоподобности;
- где линия реально объясняет выбор, а где натянута.

#### `filter`

`Специалист по продуктовой честности и обещаниям`

Может:

- запретить стратегию, если она требует слишком смелых claims;
- пометить высокий риск расхождения с отзывами.

#### `variant` по необходимости

`Креативный директор / creative strategist`

Не пишет стратегию вместе с owner.

Он нужен, если:

- есть два реально сильных стратегических хода;
- нужен альтернативный сильный вариант.

Тогда он даёт:

```text
variant B
```

А не усреднённый компромисс.

#### `approver`

Окончательный выбор между вариантами делает:

- пользователь;
- либо заранее назначенный ведущий стратег, если это когда-нибудь будет делегировано.

### Как выбираем

Сначала:

```text
что продукт реально может нести
```

Потом:

```text
какие линии из авторской матрицы этому соответствуют
```

Потом:

```text
primary + support
```

### Выход

```text
05_strategy/hero/{hero_product_key}/05__{hero_product_key}__strategy_decision.md
```

### Что должно стать понятно после этапа 5

- какое направление становится ядром стратегии;
- какое направление его поддерживает;
- какие линии отброшены;
- чем жертвуем;
- что нельзя делать дальше.

## Короткая Схема

```text
1. Intake
   выход: карта сырья + product passports + price snapshot

2. Full Signal Markup
   выход: полная разметка всех продуктов

3. Contract Matrices
   3.1 герой -> local contracts matrix
   3.2 прямые -> local contracts matrix in each competitor folder
   3.3 косвенные -> local contracts matrix in each competitor folder
   3.4 герой -> master contracts matrix

4. Matrix Conclusions
   выход: factual conclusions + interpretive hypotheses

5. Strategy Development
   выход: primary direction + supporting direction
```

## Главный Фокус До Стратегии

```text
Не правда о продукте.
Не визуал.
```
