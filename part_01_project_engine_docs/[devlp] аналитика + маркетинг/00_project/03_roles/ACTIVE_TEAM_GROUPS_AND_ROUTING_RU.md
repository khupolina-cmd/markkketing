# Активные Группы И Маршрутизация Специалистов

Статус: текущая рабочая схема.

Цель: разделить специалистов на группы, развести роли по этапам и заранее зафиксировать, кто что делает, чтобы они не передрались.

---

## 1. Как Это Устроено В Маршрутизации

Да, специалисты у нас работают как отдельные проектные инструкции и вызываются по маршруту.

Не через скиллы по умолчанию.

Базовая логика такая:

```text
этап пайплайна
→ вызывает нужные роли
→ каждая роль приходит только со своей функцией
→ итог собирает один owner
→ роль видит только разрешённые входы этапа
```

То есть не нейросеть сама решает:

- кого позвать;
- сколько людей позвать;
- кто у кого главный.

Это уже заранее прописано в маршруте.

Для реальных прогонов это не рекомендация, а операционное правило.

См.:

```text
04_methods/STAGE_ISOLATION_AND_ANTI_BIAS_PROTOCOL.md
```

## 2. Группы Специалистов

### А. Стратегическая группа

Это ядро, которое работает с матрицей и стратегией.

Сюда входят:

- `Маркетплейс-стратег / категорийный стратег`
- `Поведенческий стратег`
- `Стратег позиционирования / brand strategist`
- `Коммуникационный стратег / архитектор воронки`

Именно эта группа включается с этапа `3B`.

На этапе `3A` матрица ещё не является стратегическим выводом.

`3A` собирают технические правила, чтобы специалисты получили максимально нетронутый материал.

Начиная с `3B`, рабочие и мастер-матрицы уже не считаются чистой автоматикой.

Их собирает специалистский слой:

- маркетплейс-стратег как owner;
- поведенческий стратег как input;
- чемоданчики специалистов как обязательный источник правил.

### Б. Креативная группа

Это группа, которая подключается позже и не должна сидеть внутри матрицы.

Сюда входят:

- `Креативный директор`
- `Арт-директор`
- `Копирайтер`
- `Концепт-разработчик`

Она нужна после выбора стратегии, а не во время построения матрицы.

### В. Технические / сервисные функции

Это не решающие специалисты, а исполнители или инфраструктура.

Сюда входят:

- `AI-разметка`
- будущие скиллы;
- парсеры;
- структурирование паспортов;
- вспомогательные шаблоны.

Они не принимают смысловые решения.

## 3. Роли На Этапах

Чтобы не было драки, у каждого этапа есть не просто список людей, а тип участия.

Всего типов пять:

### `owner`

Один человек.

Пишет главный выход этапа.

### `input`

Даёт свой кусок материала.

Не собирает финальный документ.

### `filter`

Может вырезать или пометить риск только в своей границе.

### `variant`

Может дать отдельный `variant B`, если реально нужен второй сильный ход.

### `approver`

Финально выбирает между вариантами.

В нашем случае это чаще всего ты.

## 5. Как Они Коммуницируют

Не напрямую все со всеми.

А через документы этапа.

То есть логика такая:

```text
input не спорит с input
filter не переписывает весь документ
variant не замазывает owner
owner собирает итог
approver выбирает, если есть развилка
```

Это очень важно.

Иначе получится компромиссная серая каша.

## 6. Где Живут Инструкции Специалистов

Лучший вариант сейчас такой:

### 1. Одна общая маршрутизация проекта

Она отвечает на вопрос:

> кто нужен на каком этапе и зачем

Это:

```text
02_routing/READY_PIPELINE_01_05_RU.md
03_roles/ACTIVE_TEAM_GROUPS_AND_ROUTING_RU.md
```

### 2. Отдельный файл на каждого специалиста

Он отвечает на вопрос:

> как именно этот специалист думает и работает

То есть позже у каждого специалиста должен быть свой файл.

### 3. Отдельный чемоданчик специалиста

Это уже база знаний, инструменты, правила и модификаторы.

### 4. Рекомендуемая физическая структура

Чтобы потом не было хаоса по папкам, лучше держать это так:

```text
03_roles/
  ACTIVE_TEAM_GROUPS_AND_ROUTING_RU.md
  ACTIVE_TOOLKIT_FORMAT_RU.md
  ACTIVE_SPECIALIST_PROFILE_TEMPLATE_RU.md
  specialists/
    strategy_group/
      marketplace_strategist.md
      behavioral_strategist.md
      brand_strategist.md
    creative_group/
      creative_director.md
      art_director.md
      copywriter.md
      concept_developer.md
  toolkits/
    marketplace_strategist/
      default_tools.md
      user_modifiers.md
    behavioral_strategist/
      default_tools.md
      user_modifiers.md
```

То есть:

- маршрут лежит отдельно;
- личные файлы специалистов отдельно;
- чемоданчики отдельно.

## 7. Нужно Ли Дублировать Инструкции В Папке Каждого Специалиста

Да, но не всё подряд.

Логика такая:

### Общая инструкция проекта

Хранит:

- маршрут;
- порядок этапов;
- кто вызывается;
- кто owner;
- кто filter;
- кто approver.

### Личная инструкция специалиста

Хранит:

- его зону;
- его вопросы;
- его запреты;
- как он работает на разных этапах;
- как использовать чемоданчик.

То есть дублировать весь пайплайн в каждой папке не надо.

Надо дублировать только личную рабочую логику специалиста.

---

## 8. Что Оставляем Скиллами, А Что Держим Проектными Инструкциями

### Оставляем скиллами

То, что:

- повторяется часто;
- стабильно;
- не требует сложного человеческого конфликта;
- даёт один предсказуемый выход.

Сейчас сюда точно подходит:

- `signal markup`
- парсинг отзывов
- структурирование product passport из сырья

### Оставляем проектными инструкциями

То, что:

- зависит от фокуса проекта;
- требует owner / input / filter;
- меняется вместе с твоей системой;
- легко ломается, если слишком рано зацементировать в скилл.

Сейчас сюда лучше оставить:

- сборку матрицы;
- выводы по матрице;
- выбор стратегии;
- правила работы специалистов;
- их чемоданчики.

## 9. Как Это Повлияет На Время

### Если всё сразу превращать в скиллы

Минус:

- мы рано зацементируем сырой процесс;
- потом будем долго переписывать;
- модель начнёт сама слишком многое решать.

### Если всё оставить просто в голове

Минус:

- будет хаос;
- каждый раз придётся вспоминать, кто что делает;
- специалисты начнут смешиваться.

### Поэтому лучший промежуточный путь сейчас

```text
техническое = скиллы
смысловое = проектные инструкции
```

Что это даст по времени:

- старт будет быстрее, чем разрабатывать много скиллов сразу;
- пайплайн станет стабильнее;
- потом можно будет спокойно выбирать, что автоматизировать дальше.

То есть это не максимальная скорость прямо сегодня,
а лучшая скорость без развала фундамента.
