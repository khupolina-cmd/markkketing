# Канонический Словарь И Нормализация Элементов

Статус: обязательная база для `03_roles`.

Цель: чтобы специалисты, чемоданчики, инструкции и результаты говорили на одном языке, а не на смеси ролей, артефактов и случайных англицизмов.

---

## 1. Канонические Названия Ролей

| Каноническое имя | Старые / параллельные формулировки | Как использовать |
|---|---|---|
| `Стратег позиционирования` | `Руководитель позиционирования` | Так называем роль в чемоданчиках и рабочих документах этапов 4-5. |
| `Маркетплейс-стратег` | без замены | Так называем специалиста, который отличает категорийную норму от заметного отличия. |
| `Поведенческий стратег` | `Поведенческий экономист` | Так называем роль. Формулировку `поведенческая экономика` можно оставлять как название линзы, но не как имя специалиста. |
| `Коммуникационный стратег` | `Архитектор воронки` | Так называем роль. Формулировка `архитектура воронки` остается названием результата этапа 6. |
| `Креативный директор` | `Креативный стратег` только если речь про отдельный внешний input | Внутри текущей системы не смешиваем эти два названия автоматически. |
| `Арт-директор` | `Визуальный стратег` | Так называем роль. Формулировка `визуальная стратегия` остается названием типа работы. |
| `Копирайтер` | без замены | Так называем автора текстового слоя. |
| `Концепт-разработчик` | без замены | Так называем специалиста по механикам, сценкам и proof-concept ходам. |

---

## 2. Канонические Названия Элементов

- `специалист` = рабочая функция этапа, а не персонаж "вообще про всё".
- `чемоданчик` = полный набор правил специалиста.
- `дефолтные инструменты` = базовый профессиональный слой.
- `пользовательские модификаторы` = твой авторский слой поверх дефолта.
- `инструмент` = маленькая рабочая карточка из дефолтной базы.
- `модификатор` = маленькая рабочая карточка из твоих правил, которая усиливает, ослабляет или заменяет дефолт.
- `owner` = роль, которая отвечает за итоговый документ этапа.
- `input` = роль, которая подает материал, но не принимает финальное решение.
- `filter` = роль, которая ограничивает, режет и запрещает лишнее.
- `variant B` = отдельный сильный альтернативный ход, а не компромиссная смесь.

## 3. Канонические Термины Для Смысла И Доказательности

- `видимая опора` = то, что покупатель реально видит в карточке, отзывах, вопросах, цене, рейтинге, упаковке, объеме и других доступных сигналах.
- `видимый вывод` = вывод, который можно честно опереть на видимую опору.
- `интерпретация` = наше объяснение, что видимый вывод может значить для выбора.
- `человеческая правда` = не объективная истина о продукте, а живой смысл видимого факта для покупателя в конкретной ситуации.
- `обещание` = то, что карточка или коммуникация просит у покупателя принять на веру.
- `доказательство` / `proof` = то, что помогает обещанию удержаться после чтения карточки и отзывов.
- `риск обещания` = место, где обещание может сломаться после столкновения с отзывами, вопросами или реальным восприятием.
- `конфликт` = напряжение, страх, неудобство, враг категории, неловкость или контраст, который делает сообщение живым.
- `угол` = одна стратегическая ставка, а не россыпь достоинств.
- `primary` = главная ставка.
- `support` = вторая линия, которая усиливает `primary`, а не спорит с ним.
- `hold` = пока не выбрасываем, но и не несем дальше как рабочее ядро.
- `hard_reject` = жёстко убираем только дубль, невозможное, абсурд или опасную чушь.
- `claim_to_reframe` = нельзя обещать в лоб, но можно передать дальше как материал для мягкой формулировки.
- `parity_but_must_show` = не уникальность, но покупатель ждёт это увидеть как базовую норму.
- `visible_delta` = видимая разница между героем и конкурентами без выбора победителя.
- `fear_material` = материал, который может включить страх или избегание риска.
- `composition_material` = материал по составу, который может быть полезен после точечной проверки.

Главное правило:

```text
мы не называем "человеческой правдой" объективную правду о продукте;
мы называем так живой смысл того, что покупатель видит и как это считывает
```

## 4. Нормализация Формы Карточек

Во всех чемоданчиках используем только такие заголовки:

- `# Дефолтные инструменты: <роль>`
- `# Пользовательские модификаторы: <роль>`
- `## Инструмент: <название>`
- `## Модификатор: <название>`

Внутри карточек держим один и тот же набор смысловых полей:

- `Для кого`
- `Где влияет`
- `Что меняет`
- `Новое правило`
- `Почему это важно`
- `Когда применять`
- `Когда не применять`
- `Что запрещает`
- `Как меняет решение`
- `Источник`

Английский оставляем только там, где он уже работает как техническая метка:

- `owner`
- `input`
- `filter`
- `proof`
- `primary`
- `support`
- `variant B`
- `hold`
- `hard_reject`
- `claim_to_reframe`
- `parity_but_must_show`
- `visible_delta`

---

## 5. Что Больше Не Смешиваем

- не называем роль и артефакт одним словом;
- не называем гипотезу видимым фактом;
- не называем конкурентное наблюдение доказательством героя;
- не называем банальный безопасный вариант "стратегией";
- не называем красивую подачу доказательством, если она ничего не доказывает;
- не переносим social-логику на маркетплейс без поправки на среду маркетплейса.

---

## 6. Как Читать Книги И Переносить Их В Систему

Если книга говорит про "truth", "voice", "platform", "tension", "idea" или "brand behavior", мы сначала переводим это в канонический язык системы:

- что здесь является `видимой опорой`;
- где здесь `человеческая правда`;
- что становится `обещанием`;
- где `доказательство`;
- какой специалист этим управляет;
- на каком этапе это вообще разрешено.

Только после этого мысль можно класть в чемоданчик.
