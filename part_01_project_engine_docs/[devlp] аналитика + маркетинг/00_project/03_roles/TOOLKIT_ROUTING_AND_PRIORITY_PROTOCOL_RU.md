# Протокол Маршрутизации И Приоритизации Чемоданчиков

Статус: активное правило для всех специалистов.

Цель: чтобы специалист не листал весь чемоданчик как меню, а понимал:

- что у него включается всегда;
- что включается по триггеру;
- что подключается только для усиления;
- что нужно только для защиты сильной идеи.

---

## 1. Главное Правило

```text
специалист не выбирает "из всего интересного"
специалист идет по маршруту:
этап -> задача этапа -> тип ситуации -> 1 обязательный модификатор -> 1-2 триггерных модификатора
```

То есть чемоданчик не работает как библиотека "на всякий случай".

Он работает как узкий набор переключателей.

---

## 2. Четыре Типа Включения

У каждого модификатора должен быть один из четырех типов включения:

### 1. `always`

Это обязательный фильтр специалиста на этом этапе.

Он включается почти всегда, если специалист вообще вызван маршрутом.

Пример:

- `сначала раздели видимую опору и интерпретацию`;
- `проверь человеческий голос`;
- `один слайд = одна работа`.

### 2. `trigger`

Это модификатор по ситуации.

Он включается только если есть конкретный признак:

- категория шумная;
- идея банальная;
- экран перегружен;
- концепция слишком умная;
- команда боится сильного хода.

### 3. `amplifier`

Это усилитель.

Он не нужен всегда.

Он нужен, когда базовая линия уже выбрана и надо сделать ее:

- острее;
- чище;
- заметнее;
- более масштабируемой.

### 4. `defense`

Это защитный модификатор.

Он нужен, когда:

- сильную идею надо объяснить;
- нестандартный ход могут зарезать из страха;
- нужно показать, почему смелая линия безопаснее серой альтернативы.

---

## 3. Порядок Выбора Для Любого Специалиста

### Шаг 1. Маршрут уже выбрал специалиста

Сам специалист не решает, нужен ли он вообще.

Это делает этап.

### Шаг 2. Специалист включает один `always`

Это его главный фильтр чтения задачи.

Без него он вообще не начинает думать.

### Шаг 3. Специалист ищет один доминирующий триггер

Не пять.

Один главный.

Например:

- категория шумная;
- идея банальная;
- экран не считывается;
- текст звучит фальшиво.

Под этот триггер он берет один `trigger`.

### Шаг 4. При необходимости берет один `amplifier`

Только если базовая линия уже выбрана и ее надо усилить.

### Шаг 5. При конфликте или защите идеи берет один `defense`

Это уже не для генерации, а для объяснения и удержания решения.

---

## 4. Жесткий Лимит

На одну задачу специалист обычно берет:

- `1 always`
- `1 trigger`
- иногда `+1 amplifier` или `+1 defense`

Нормальный потолок:

```text
2-3 модификатора на один проход
4 только если задача реально сложная
```

Если специалисту "нужно" 5-6 модификаторов сразу,
значит чемоданчик собран плохо или задача не разложена.

---

## 5. Как Определять Приоритет

Приоритет ставим не по красоте мысли, а по силе влияния на решение.

### `P1`

Без этого модификатора специалист системно ошибается.

### `P2`

Без него решение будет рабочим, но более банальным, рыхлым или шумным.

### `P3`

Это полезное усиление, но не ядро решения.

---

## 6. Как Это Выглядит В Чемоданчике

Сверху каждого `user_modifiers.md` должен быть короткий блок:

- `Маршрут выбора`
- `Карта модификаторов`

В карте фиксируем:

- название модификатора;
- тип включения;
- приоритет;
- когда включать;
- когда он идет по порядку.

---

## 7. Красные Флаги

- специалист начинает "вдохновляться" всем списком;
- включает два `always` одновременно без причины;
- берет `defense`, когда еще не выбрана линия;
- берет `amplifier`, когда базовая логика еще не собрана;
- прыгает сразу к любимому модификатору и пропускает базовый фильтр.

---

## 8. Короткая Формула

```text
не "какой модификатор мне нравится"
а
"какой у задачи сейчас тип, и какой следующий правильный переключатель"
```
