# Copy Expansion Before Modifiers

## Зачем нужен этот документ

Зафиксировать рабочую логику копирайтингового слоя до модификаторов.

Ключевая идея:

- мы не схлопываем стратегию сразу в "безопасную" формулировку;
- мы сначала расширяем массив текстовых вариантов;
- потом отбираем живую базу;
- и только потом прогоняем выжившее через модификаторы.

## Что уже зафиксировано

### 0. Иерархия слоев

Важно не смешивать слои между собой.

Каждый следующий слой модифицирует предыдущий, а не дублирует его работу.

Рабочая иерархия сейчас такая:

1. `Смысловой слой`
Что должен сделать слайд и что должен сделать конкретный элемент.

2. `Слой операторов расширения`
Через какие призмы мы раскрываем один и тот же смысл.

3. `Слой отбора`
Что из массива остается живым и годным для дальнейшей обработки.

4. `Слой модификаторов`
Какой голос, ритм и стилистический режим получает уже выбранная база.

5. `Proof-слой`
Подключается только там, где реально нужна внешняя опора.

Это значит:

- стилевые свойства не нужно дублировать до модификатора;
- операторы расширения не должны подменять модификаторы;
- proof не должен управлять функцией слайда;
- один и тот же вопрос не должен решаться в двух слоях сразу.

### 1. Модификаторы не трогаем

Модификаторы уже работают как отдельный слой.

Их задача:

- не искать новые идеи;
- не строить архитектуру слайда;
- не решать, что именно должен делать слайд;
- а усиливать качество речи, фактуру, чувствительность и голос на уже подготовленном массиве.

### 2. Сначала функция слайда, потом доказательство

Обычный порядок такой:

1. понять, зачем нужен слайд;
2. понять, какими ходами он может выполнить эту функцию;
3. только потом, если нужно, искать доказательную опору.

Доказательство - это роскошь, а не базовое условие каждого слайда.

Обычно proof-слой нужен только там, где без опоры слайд выглядит пустым или слишком общим.
Практически это чаще всего `1-2` слайда на карточку.

### 3. Жесткой ручной разметки текста заранее не делаем

Мы не обязаны заранее вручную писать "черновой заголовок", "черновой подзаголовок" и "черновой основной текст".

Вместо этого мы сначала задаем функциональную микроспеку элемента.

Текстовая форма может рождаться уже внутри креативной генерации.

### 4. Единица креативной генерации - отдельный текстовый элемент

Креативная генерация работает не над слайдом целиком, а по одной единице:

- `заголовок`;
- `подзаголовок`;
- `основной текст`;
- при необходимости `микрокопия`.

### 5. Стилевые теги задаются на уровень проекта или всей карточки

Не нужно дублировать одинаковые стилевые теги на каждый отдельный элемент.

Если у всей карточки одна стилевая доминанта, она фиксируется один раз на уровне проекта или карточки.

Примеры таких тегов:

- более сдержанно;
- мягко;
- без пафоса;
- с легкой сенсорикой;
- с иронией;
- без юмора;
- предметно;
- без luxury-кода.

Если эти свойства уже сидят внутри выбранного модификатора, не нужно дополнительно прописывать их в операторе расширения.

Иначе получится бессмысленное дублирование одного и того же свойства на двух этажах системы.

### 6. Реестр ходов единый, но живой

Набор допустимых ходов:

- не собирается с нуля под каждый проект;
- живет как общий реестр;
- дополняется, если в проекте не хватает существующих решений.

Это допустимая шаблонность, потому что решения становятся собственными и осмысленными, а не рыночным мусором по умолчанию.

## Минимальный паспорт текстового элемента

Перед генерацией у каждого элемента должен быть минимальный функциональный паспорт.

Обязательный минимум:

- `element_type`: заголовок / подзаголовок / основной текст / микрокопия;
- `slide_function`: что делает слайд в воронке;
- `element_job`: что должен сделать именно этот элемент;
- `core_meaning`: какой смысл он обязан донести;
- `reader_effect`: что человек должен почувствовать или понять после чтения;
- `forbidden_effect`: чего этот элемент не должен делать;
- `allowed_moves`: какими ходами ему разрешено решать задачу.

Дополнительно, если нужно:

- `tone_limit`: что нельзя по тону;
- `proof_needed`: да / нет;
- `examples_to_touch`: о чем можно говорить предметно;
- `examples_to_avoid`: куда нельзя уводить текст.

## Что такое allowed moves

`allowed_moves` - это не готовые фразы, а разрешенные способы выполнить функцию.

Примеры:

- через механику;
- через формулу;
- через краткосрочный эффект;
- через долгосрочный эффект;
- через инверсию;
- через сенсорику;
- через трение;
- через контраст;
- через everyday-сценарий;
- через снятие страха;
- через честный self-selection;
- через mood-frame;
- через дорожную карту состояний.

Именно здесь фиксируется, как слайд может быть небанальным без потери смысла.

Это именно операторы расширения, а не модификаторы.

Их задача:

- не задавать стиль автора;
- не делать текст "нежным", "ироничным" или "доверительным";
- а раскрывать одну и ту же единицу материала под разными углами.

Рабочее правило для тестов:

- `1` оператор расширения может отдавать до `6` единиц материала на один элемент;
- один элемент можно прогонять через несколько операторов подряд;
- итоговый объем зависит от задачи, но не должен разрастаться бесконтрольно.

## Рабочий порядок до модификаторов

1. Зафиксировать общую стилевую доминанту проекта или карточки.
2. Для каждого слайда записать его функцию.
3. Для каждого текстового элемента внутри слайда собрать минимальный паспорт.
4. Для каждого элемента выбрать допустимые ходы из общего реестра.
5. Запустить креативную генерацию по одному элементу за раз.
6. Получить массив сырых, местами кривых, но смыслово живых вариантов.
7. Отсечь мусор и оставить базовый массив выживших формулировок.
8. Только после этого прогнать выжившее через модификаторы.
9. Для `1-2` proof-слайдов отдельно подключить доказательные опоры, если они действительно нужны.

### Что куда относится

`Смысловой слой`

- функция слайда;
- работа элемента;
- основной смысл;
- эффект на читателя;
- запреты.

`Операторы расширения`

- сенсорика;
- трение;
- жизненная сцена;
- продукт-герой;
- механика;
- формула;
- краткосрочный / долгосрочный эффект;
- инверсия;
- contrast / mood-frame / self-selection и другие приемы раскрытия.

`Модификаторы`

- ясность и доверие;
- нежность;
- спокойная мобилизация или уверенная активация;
- ирония;
- авторские или экспериментальные модификаторы вроде Рори Сазерленда.

Главный принцип:

сначала мы ищем разные углы раскрытия,
а потом выбираем, в каком голосе эти углы должны заговорить.

## Что является хорошим результатом до модификаторов

На этом этапе текст еще может быть неровным.

Это нормально.

Хороший промежуточный результат выглядит так:

- стратегия не повторяется буквально;
- есть несколько реально разных ходов, а не десять перестановок одних и тех же слов;
- смысл сохраняется;
- текст не уходит в чужую задачу;
- среди вариантов уже есть за что цепляться;
- массив можно улучшать модификатором, а не спасать с нуля.

## Первый тест

### Цель теста

Проверить, дает ли такая схема более живой массив до модификаторов, чем прямой переход от стратегии к готовому копирайтингу.

### Что тестируем первым

Не всю карточку.

Берем:

- `1` обложку;
- `1` информационный слайд;
- при желании `1` proof-слайд как отдельный дополнительный тест.

### Шаги первого теста

1. Взять один реальный продукт и одну уже выбранную стратегию.
2. Зафиксировать стилевые теги проекта на уровне всей карточки.
3. Для обложки заполнить минимальные паспорта хотя бы для `headline`, `subhead` и `microcopy` или `body`, если он нужен.
4. Для каждого элемента выбрать `2-3` допустимых хода из общего реестра.
5. Запустить креативную генерацию отдельно по каждому элементу.
6. Для обложки собрать не меньше `10` вариантов на элемент.
7. Для обычного информационного слайда собрать суммарно около `10` вариантов на элемент или на один короткий проход по каждому ходу.
8. Удалить мусор и оставить короткий массив выживших формулировок.
9. Прогнать выжившее через один стабильный модификатор.
10. При желании прогнать ту же базу через один экспериментальный модификатор и сравнить.
11. Для proof-слайда отдельно проверить, нужен ли ему внешний proof вообще.
12. Если нужен, добавить только безопасную заимствованную опору без вранья про сам продукт.

### Что сравниваем в тесте

Сравниваем не "красоту вообще", а вот это:

- стало ли меньше буквального повторения стратегической формулы;
- появились ли реально разные способы подать одну и ту же мысль;
- стало ли больше удачных фраз до ручного отбора;
- помогает ли модификатор усиливать уже живую базу, а не маскировать пустоту;
- не ломается ли честность claims после подключения proof-слоя.

## Самый короткий вывод

Копирайтинг в этом пайплайне начинается не с написания готового текста, а с правильной постановки задачи для элемента.

Сначала:

- функция;
- работа элемента;
- эффект на читателя;
- допустимые ходы.

Потом:

- креативное расширение;
- отбор;
- модификаторы;
- proof только там, где он правда нужен.
