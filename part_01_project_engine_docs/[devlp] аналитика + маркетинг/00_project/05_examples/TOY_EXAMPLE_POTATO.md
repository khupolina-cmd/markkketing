# Toy Example: Potato Pipeline

Цель: отладить логику пайплайна на максимально банальном предмете, чтобы не путаться в косметике, маркетплейсах и красивых терминах.

Предмет: картошка.

Важно:

```text
это ранний упрощённый логический пример
по актуальной маршрутизации и ролям главным источником теперь считаем:
TOY_POTATO_PIPELINE_TO_STAGE_4.md
TOY_POTATO_ROLE_RUN_03_05.md
```

---

## 1. Сырые отзывы

```text
1. Картошка вкусная, легко чистится, беру для пюре.
2. Попадается мелкая, но для супа нормально.
3. Быстро разваривается, для жарки не подошла.
4. Грязная, долго мыть, больше не возьму.
```

---

## 2. Этап 02: AI Signal Markup

Этот этап делает AI без стратегических специалистов.

| id | отзыв | сигналы |
|---:|---|---|
| 1 | Картошка вкусная, легко чистится, беру для пюре. | плюс: вкусная; плюс: легко чистится; сценарий: пюре |
| 2 | Попадается мелкая, но для супа нормально. | минус: мелкая; сценарий: суп; терпимость: нормально для супа |
| 3 | Быстро разваривается, для жарки не подошла. | плюс/свойство: быстро разваривается; сценарий: жарка; стоп: не подошла для жарки |
| 4 | Грязная, долго мыть, больше не возьму. | негатив: грязная; негатив: долго мыть; стоп-интерпретация: больше не возьму |

Что важно:

AI только раскладывает отзывы на сигналы.

AI пока не решает:

- для кого картошка;
- какой у неё угол;
- как её продавать;
- какие конкуренты.

---

## 3. Этап 03: Diagnostic Matrix

Это тот самый спорный этап после разметки.

В актуальной версии маршрута это не отдельный герой-специалист,
а ранний сжатый прототип того, что потом стало этапом `03 Contract Matrices`.

То есть по смыслу здесь уже видна та же логика:

- не делать огромные призмы;
- не склеивать всё насильно;
- собирать контракт только там, где связь реально держится.

Формат: одна компактная матрица.

| Наблюдение | Сигнальная база | Минус | Кто терпит / не терпит | Сценарий / ожидание | Интерпретационная заметка | Стоп-фактор | Можно собрать сделку? |
|---|---|---|---|---|---|---|---|
| Картошка мелкая, но для супа нормально | негатив + сценарий | мелкая | терпят те, кто варит суп | суп | приготовить суп, где размер не критичен | если нужна крупная картошка | Да: "беру для супа, мелкость не мешает" |
| Быстро разваривается, для жарки не подошла | позитив/свойство + негатив + сценарий | разваривается | терпят / любят для пюре, не терпят для жарки | пюре vs жарка | получить мягкую картошку / сохранить форму кусочков | не держит форму при жарке | Да: "хороша для пюре, плоха для жарки" |
| Грязная, долго мыть | негатив + отказ | грязная | часть может терпеть ради вкуса/цены, но в отзыве не терпят | любое использование | не тратить время на мойку | больше не возьму | Нет: это стоп без видимой терпимости |
| Вкусная и легко чистится | позитив + сценарий | нет явного минуса | не нужно | пюре | быстро приготовить вкусное пюре | не найден | Нет: это плюс + сценарий, но не сделка через минус |

---

## 4. Что Получилось

Мы не делали три отдельных пайплайна:

- отдельно от минуса;
- отдельно от сценария;
- отдельно от сценария / ожидания.

Мы сделали одну таблицу, где они стоят рядом.

Но мы не склеивали их насильно.

Сделка появляется только там, где есть реальная связка:

```text
покупаю ради X -> терплю Y -> использую в Z -> откажусь при C
```

---

## 5. Мини-вывод по картошке

### Сильные диагностические сделки

1. `Картошка для супа`

- покупают / принимают для супа;
- мелкость не критична;
- риск: не подойдёт тем, кто хочет крупные клубни.

2. `Картошка для пюре`

- быстро разваривается;
- это плюс для пюре;
- это же минус для жарки.

### Стоп-факторы

1. `Грязная, долго мыть`

- это не просто минус, а отказ в отзыве 4;
- пока не видно группы, которая это терпит.

2. `Не держит форму для жарки`

- стоп для сценария жарки;
- не стоп для пюре.

---

## 6. Почему Это Лучше, Чем Сразу "Призмы"

Если бы мы делали призмы, могли бы получить:

- "Любители пюре";
- "Суповой сценарий";
- "Не наша аудитория: жарка";
- "Операционный барьер: грязь".

Это уже похоже на мини-отчёт.

Но для второго этапа достаточно проще:

```text
свойство -> где плюс -> где минус -> кому подходит -> где стоп
```

---

## 7. Метод Для Реального Товара

После signal markup героя делаем такую же матрицу:

| Наблюдение | Минус | Кто терпит / не терпит | Сценарий | Боль / задача | Стоп-фактор | Можно собрать сделку? |
|---|---|---|---|---|---|---|
| ... | ... | ... | ... | ... | ... | да / нет |

Ограничение:

не больше 10-12 строк на первом проходе.

Если строк больше, значит мы снова начали писать отчёт, а не диагностическую матрицу.

---

## 8. Роли На Игрушечном Примере

### AI-разметчик

Разметил отзывы на сигналы.

### Сжатый прототип этапа 3

Здесь в мини-форме уже виден будущий этап `Contract Matrices`.

### Поведенческий экономист

Добавил бы только один комментарий:

быстро разваривается = не объективный плюс или минус. Это зависит от сценария. Для пюре плюс, для жарки минус.

### Граница обещаний

На этом месте система всё равно запретила бы писать:

- "идеальна для всего";
- "подходит для жарки";
- "чистая картошка".

### Руководитель позиционирования

Если бы нужно было продавать, выбрал бы:

`для пюре и супа`, а не `универсальная картошка`.

Но это уже следующий этап, не диагностика.

---

## 9. Главный Урок

Нам не нужно выбирать между:

- разбивкой по минусу;
- разбивкой по сценарию;
- разбивкой по боли;
- сделкой.

Правильнее:

1. сначала держать минус, сценарий / ожидание и стоп-фактор отдельными колонками;
2. смотреть, где между ними есть реальная связь;
3. только там собирать "сделку";
4. всё, что не связано, оставлять как отдельное наблюдение или стоп-фактор.

Так система остаётся умной, но не раздувается.
