# Сборщик Product Passport

Используй этот skill, чтобы превращать сырое и неаккуратное сырьё карточки товара в один готовый `product_passport` в markdown.

Этот skill не управляет папками, маршрутом проекта и этапами пайплайна. Его задача только одна: вернуть содержимое паспорта и предложить имя файла.

Перед работой прочитай:

- [references/input-template.md](references/input-template.md)
- [references/output-template.md](references/output-template.md)

## Что относится к этому skill

- скопированный текст со страницы товара
- описание селлера
- характеристики и спецификации
- цена, рейтинг, число отзывов, число покупок, если видно
- грубые голосовые заметки с текстом со слайдов или инфографики
- частичные факты, которые нужно привести в порядок

## Что сюда не относится

- парсинг отзывов
- signal markup
- позиционирование
- сравнение с конкурентами
- критика карточки
- вкусовые суждения о визуале

Если пользователь начинает просить что-то из этого списка, аккуратно закончи сборку паспорта и передай работу следующему skill или в другой чат, а не растягивай этот skill на чужие задачи.

## Минимальные блокирующие поля

Это единственные поля, из-за отсутствия которых можно остановиться и задать короткий вопрос:

```text
product_key:
product_role: hero / direct / indirect
marketplace:
```

Если чего-то из этого нет и это нельзя безопасно вывести из контекста, задай один короткий вопрос.

Всё остальное можно заполнять как `unclear`.

## Язык ответа

- Общайся с пользователем только по-русски.
- Не переходи на английский в объяснениях, комментариях и служебном тексте.
- Исключения: канонические имена полей, `product_key`, имя файла и фиксированные значения шаблона, если они уже заданы стандартом проекта.

## Порядок работы

1. Сначала один раз внимательно прочитай всё сырое сырьё пользователя.
2. Мысленно приведи свободный ввод к форме из [references/input-template.md](references/input-template.md).
3. Вытащи только факты и claims селлера.
4. Преврати грубые голосовые заметки по инфографике в `image_text_claims`.
5. Если важна точная формулировка обещания, держись как можно ближе к словам пользователя.
6. Убери дубли между копипастой со страницы и голосовыми заметками.
7. Отдели факты от предположений.
8. Собери финальный паспорт, используя точные имена полей из [references/output-template.md](references/output-template.md).
9. Всё сомнительное и неполное складывай в `missing_or_unclear` и `needs_manual_check`.
10. Держи результат готовым для будущего `price_value_snapshot`, но не пытайся создавать этот файл внутри этого skill.

## Правила

- Не критикуй дизайн карточки.
- Не анализируй визуальный стиль.
- Не описывай первое впечатление.
- Не сравнивай продукт с конкурентами.
- Не превращай claims в стратегию.
- Не придумывай отсутствующие факты.
- Если что-то не видно или не сказано явно, пиши `unclear`.
- Считай обещания селлера claims, а не доказанной истиной.
- Считай продиктованный текст со слайдов фактическим вводом о том, что обещает карточка.
- Игнорируй чисто эстетические комментарии вроде `выглядит дорого` или `нежный визуал`.
- Если эстетический комментарий может скрывать фактическое обещание, сохрани только фактическую часть и пометь неоднозначность в `needs_manual_check`.

## Как обрабатывать голосовые заметки

Голосовые заметки по умолчанию считаются сырыми и неточными.

Нормализуй их так:

- исправляй очевидный шум распознавания только если смысл и так ясен
- держи важные формулировки claims как можно ближе к источнику
- если фраза остаётся двусмысленной, сохрани очищенную версию и пометь неоднозначность в `needs_manual_check`
- если пользователь говорит `на слайде написано...`, считай это claim
- если пользователь только описывает визуал без текста, не запихивай это насильно в паспорт

## Формат ответа

Формат ответа по умолчанию:

1. Одна строка:

```text
Рекомендуемое имя файла: 01__{product_key}__product_passport.md
```

2. Потом один fenced `md` block, внутри которого лежит только готовое содержимое файла.

3. После блока не пиши длинных объяснений.

Если пользователь явно просит plain markdown без блока, верни plain markdown.

Если не хватает минимальных блокирующих полей, задай один короткий вопрос вместо поломанного результата.

## Планка качества

Паспорт должен быть полезным как ранний фактический слой для дальнейшей матрицы.

Это значит:

- достаточно коротким, чтобы его было легко читать
- достаточно полным, чтобы он фиксировал claims и базу сравнения
- достаточно строгим, чтобы другой чат мог доверять именам полей
- достаточно честным, чтобы отмечать неопределённость вместо выдумки

## Примеры триггеров

- `собери паспорт товара из карточки`
- `я сейчас вставлю всё с сайта и продиктую инфографику`
- `сделай product passport`
- `структурируй карточку товара в паспорт`
- `из этого сырья нужен готовый паспорт героя`
