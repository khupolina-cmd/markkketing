## Причина Проверки

Пользовательская правка: система начала звучать так, будто ищет объективную правду, подтверждение и опровержение.

Правильный фокус проекта другой:

```text
мы смотрим, что человек увидит в карточке, фото и отзывах,
как он это прочитает,
и как будущая воронка должна не спорить с этим восприятием
```

Дополнительное уточнение:

```text
отзывы не являются правдой о продукте
отзывы являются видимыми социальными сигналами
```

Покупатель может поверить отзыву, испугаться, отмахнуться, решить что отзыв куплен, или посчитать автора странным.

Система должна учитывать не истинность отзыва, а его возможный вес в решении.

## Главная Поправка

`confirmation` больше не читается как лабораторное доказательство.

В этой системе это означает:

```text
линия выглядит безопасной и знакомой для покупательского чтения
```

Слабая видимая опора не равна отказу.

Она может означать:

- `angle_candidate`;
- осторожный сценарий;
- настрой ожиданий;
- support-only аргумент;
- запрет на прямой claim.

## Что Исправлено

| Зона | Был риск | Исправление |
|---|---|---|
| Core focus | "подтверждение" звучало как проверка правды продукта | переопределено как безопасность восприятия |
| Stage 3 method | `hold` звучал как отказ | добавлен `angle_candidate`, `hold` описан как парковка |
| Technical draft | мог читать строки как продуктовую правду | закреплено: 3A фиксирует только видимые данные |
| Working/master matrix | travel-сценарий был в `hold` | переведён в `angle_candidate` |
| Dashboard | специалистский блок говорил языком `confirmed / hold / reject` | добавлен язык perception-ready и `angle_candidate` |
| Review logic | отзывы могли читаться как источник истины | закреплено: отзывы = социальные сигналы, а не истина |

## Travel / Сумка / До 100 Мл

Почему это больше не `hold`:

- человек видит объём 75 мл;
- в отзывах героя есть сценарий "в сумке / с собой";
- формат меньше больших body-конкурентов;
- карточка уже продаёт продукт как визуальный объект, который приятно иметь при себе;
- этот угол может работать через восприятие, даже если он не является массовым отзывным сценарием.

Правильный статус:

```text
angle_candidate
```

Правильная граница:

```text
не обещать "идеально для самолёта" как массовый use-case
можно проверять фрейм "с собой / на стол / в поездку"
```

## Проверка По Этапам

| Этап | Фокус после правки |
|---|---|
| 01 Intake | фиксирует сырьё и видимые данные, не интерпретирует стратегию |
| 01.5 Passports + Price | фиксирует то, что покупатель или система видит одинаково |
| 02 Signal Markup | разбирает отзывы на видимые сигналы и evidence IDs, не выбирает стратегию |
| 03A Technical Draft | собирает широкий кандидатник, не ищет "правду продукта" |
| 03B Working Matrices | переводит видимые данные в покупательское чтение |
| 03C Master Matrix | собирает perception-ready линии, claim risks и angle candidates |

## Новое Правило Для Следующего Шага

Этап 4 должен брать из мастер-матрицы не "истины", а:

- что покупатель увидит;
- что он может считывать как норму;
- где у него появится тревога;
- какой фрейм делает минус терпимым;
- какие `angle_candidate` стоит передать в стратегическое обсуждение.

Коротко:

```text
не ищем правду
строим совпадение ожидания карточки с тем, что человек увидит в отзывах
```
