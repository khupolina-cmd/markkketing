# Промпт 04: Детекция и классификация CLAIM_CONFLICT

## Назначение

Для каждого обещания с вердиктом 🔴 или ⛔ из `promise_safety_map` — классифицировать тип конфликта, сформулировать предупреждение для заказчика и предложить варианты действий.

---

## Входные данные

```
promise_safety_map: [из промпта 03]
producer_claims: [список обещаний производителя]
client_brief: [если предоставлен заказчиком — опционально]
```

---

## Инструкция

### Шаг 1: Отфильтровать конфликтные обещания

Из `promise_safety_map` взять только обещания с вердиктом **🔴 (Опасно)** или **⛔ (Запрещено)**.

Если таких нет → выдать финальный статус «расхождений не обнаружено» и завершить.

### Шаг 2: Для каждого конфликтного обещания — определить тип

Прочитать описание и опровержения. Сопоставить с 5 типами конфликтов:

| Тип | Признаки |
|---|---|
| `cosmetic_gap` | Обещание чуть завышено, продукт в целом работает, расхождение небольшое |
| `perception_gap` | Продукт работает, но покупатели ожидают большего из-за формулировки |
| `functional_gap` | Значимая доля (~25–40%) не получает обещанного; остальные довольны |
| `fundamental_gap` | Большинство (>50%) не получают обещанного; проблема продуктовая |
| `safety_gap` | Есть сигналы вреда здоровью, аллергии, опасности |

Если обещание соответствует нескольким типам → выбрать **наиболее серьёзный** (safety_gap > fundamental_gap > functional_gap > perception_gap > cosmetic_gap).

### Шаг 3: Определить severity конфликта

- `cosmetic_gap` → low–medium
- `perception_gap` → medium
- `functional_gap` → high
- `fundamental_gap` → critical (для обещания)
- `safety_gap` → critical (безусловный)

### Шаг 4: Сформулировать предупреждение для заказчика

Использовать формулировки из [claim-conflict-protocol.md](../references/claim-conflict-protocol.md).

Тон: информирующий, не паникующий, но чёткий. Предупреждение должно быть невозможно проигнорировать.

### Шаг 5: Предложить варианты действий

Для каждого конфликта — 2–3 конкретных варианта действий с карточкой.

Примеры вариантов:
- Смягчить формулировку (для cosmetic_gap)
- Добавить контекст/условие (для perception_gap)
- Убрать из центра, упомянуть вторично (для functional_gap)
- Убрать полностью (для fundamental_gap)
- Добавить дисклеймер + согласовать с юристом (для safety_gap)

---

## Формат выхода

### Если конфликтов нет:

```
=== CLAIM_CONFLICT_REPORT ===
Статус: Расхождений не обнаружено.
Все обещания производителя имеют вердикт ✅, 🟡 или ⚠️ — в пределах допустимого.
```

### Если конфликты есть:

```
=== CLAIM_CONFLICT_REPORT ===
Обнаружено конфликтов: N

=== CLAIM_CONFLICT #1 ===
Обещание: «[текст]»
Источник: [упаковка / бриф / карточка]
Тип конфликта: [тип]
Severity конфликта: [уровень]

Опровергающие сигналы:
- «[тема]»: N упоминаний, severity = [уровень]

Поддерживающие сигналы:
- «[тема]»: N упоминаний
[Или: поддерживающих сигналов нет]

⚠️ ПРЕДУПРЕЖДЕНИЕ ДЛЯ ЗАКАЗЧИКА:
[Текст предупреждения]

Варианты действий:
1. [вариант A]
2. [вариант B]
3. [вариант C]

---

[Повторить для каждого конфликта]
```

---

## Правила и проверки

- [ ] Проверены все обещания с вердиктом 🔴 и ⛔ из promise_safety_map
- [ ] Тип конфликта определён для каждого
- [ ] safety_gap отмечен максимально явно
- [ ] Варианты действий конкретные, не общие
- [ ] Если конфликтов нет — честно зафиксировано

---

## Анти-галлюцинаторные правила

1. Не классифицировать как конфликт обещания с вердиктом ✅, 🟡, ⚠️
2. Не решать за заказчика — только информировать и предупреждать
3. Не смягчать severity safety_gap — это всегда critical
4. Не игнорировать client_brief: если заказчик настаивает на спорном обещании — зафиксировать это явно в предупреждении
