# Skill: creative-angle-discovery

## Purpose

Используй этот скилл, когда у пользователя уже есть результат по герою и библиотека косвенных конкурентов, а задача — найти неочевидные углы позиционирования, которые не подтверждены напрямую отзывами героя, но могут быть физически и маркетингово правдоподобны.

Это не скилл для подтверждения существующих призм. Для прямого усиления уже найденных призм есть отдельный навык `prism-reinforcement`.

## Use When

- Есть `perception-prisms-[герой].md` по товару-герою.
- Есть `signal-markup-[герой].md`.
- Есть `prisms-library-indirect.md` по косвенным конкурентам.
- Есть `product-context-[герой].md`.
- Пользователь хочет найти новые creative angles, экспериментальные призмы или «диагональные» углы.

## Do Not Use When

- Нужно просто подтвердить или усилить уже существующие призмы героя.
- Есть только прямые конкуренты.
- Нет контекста продукта, и из-за этого нельзя проверить физическую правдоподобность углов.

## Inputs

Нужны все четыре документа:

1. `perception-prisms-[герой].md`
2. `prisms-library-indirect.md`
3. `signal-markup-[герой].md`
4. `product-context-[герой].md`

## Core Logic

Главный принцип скилла: `нет данных` не равно `не подходит`.

Этот навык обязан делать двухэтапную фильтрацию:

1. Проверка по сигналам героя.
2. Проверка по контексту продукта.

Если угол не подтверждается сигналами героя, его нельзя автоматически отсечь. Нужно проверить, физически возможен ли он для продукта героя, и только потом принимать решение.

## Required Workflow

### Step 0. Inventory

- Прочитай `prisms-library-indirect.md` целиком.
- Зафиксируй всех косвенных конкурентов и их кластеры.
- Прочитай `perception-prisms-[герой].md` и составь список уже существующих призм героя.
- Прочитай `product-context-[герой].md` и зафиксируй ограничения продукта: состав, способ применения, упаковку, формат, инфографику, сегмент.

### Step 1. Fast Deduplication

Для каждого кластера косвенного конкурента:

- Если у него совпадают не меньше двух драйверов с уже существующей призмой героя, помести его в блок подтверждений.
- Если не совпадает, передай дальше на фильтрацию.

### Step 2. First Filter: Signals

Для каждого не совпавшего кластера:

- Ищи хотя бы один похожий сигнал в `signal-markup-[герой].md`.
- Если следы у героя есть, оформляй это как экспериментальную призму.
- Если следов нет, не отсекай: передай в следующий шаг.

### Step 3. Second Filter: Product Context

Если у героя нет прямых сигналов, проверь угол через `product-context-[герой].md`.

Используй все пять техник:

1. Инверсия сценария.
2. Аудиторный мост.
3. Компромиссный мост.
4. Слепое пятно выдачи.
5. Формат-хакинг.

Если хотя бы одна техника даёт правдоподобный угол, оформляй его как `creative-angle` с пометкой, какой техникой он найден.

Если ни одна техника не сработала, отправляй элемент в отсечённое и объясняй причину.

### Step 4. Final Filter

Каждый найденный угол обязан пройти финальную проверку:

1. Не противоречит `product-context`.
2. Не ломает `Conflict Map` героя.
3. Не нарушает секцию «что нельзя утверждать».
4. Выглядит уместно для карточки и реального использования продукта.

## Output Contract

Вывод должен быть разбит минимум на 4 блока:

1. Подтверждения.
2. Экспериментальные призмы.
3. Creative angles.
4. Отсечённое.

Дополнительно полезно выводить жизненный цикл документов, если пользователь работает всей цепочкой.

## Confidence Rules

- `●` — сильная уверенность.
- `◐` — умеренная уверенность.
- `○` — слабый сигнал или тестовый угол.

Все `creative-angle` по умолчанию получают `○`, потому что они не подтверждены прямыми сигналами героя.

## Hard Rules

- Не использовать проценты, только количество отзывов.
- Не называть отсутствующие данные доказательством несовместимости.
- Для каждого `creative-angle` обязательно указывать технику, через которую он найден.
- Не придумывать медицинские, функциональные или продуктовые обещания, которых нет в данных.
- Не подменять creative angle подтверждённой призмой и наоборот.

## Response Style

- Пиши ясно и прикладно.
- Для каждого найденного угла показывай источник, логику и ограничение.
- Обязательно разделяй: что подтверждено, что экспериментально, что только на тест.
