# Работа с конкурентами — пошаговая инструкция

Этот документ описывает, как использовать Skill 2 для анализа конкурентов и накопления библиотеки кластеров. Инструкция рассчитана на человека без маркетингового образования.

---

## Зачем это нужно

После первого прогона Skill 2 по своему товару у тебя есть гипотезы: «вот наши кластеры покупателей». Но это только наши данные. Мы не знаем, универсальны ли эти кластеры для всей категории или специфичны именно для нашего товара.

Прогон по конкурентам даёт ответ:
- **Прямые конкуренты** → **ВАЛИДАЦИЯ**: подтверждаются ли наши кластеры у других?
- **Косвенные конкуренты** → **РАСШИРЕНИЕ**: есть ли пучки факторов, которых у нас нет?

---

## Работа с прямыми конкурентами (ВАЛИДАЦИЯ)

### ШАГ 1: ВЫБОР КОНКУРЕНТА

Критерии выбора:
- Тот же тип товара (например, парфюмированный крем для рук)
- Похожий ценовой сегмент (±30% от цены нашего товара)
- Достаточно отзывов (50+)
- Не копия нашего бренда (другой производитель)

Сколько конкурентов: **2–3 прямых конкурента** достаточно для валидации.

### ШАГ 2: ПРОГОН SKILL 1 (signal-markup)

Скачай отзывы конкурента → прогони через Skill 1 (signal-markup).

Результат сохрани как: `[название-конкурента]-signal-markup.md`

### ШАГ 3: ПРОГОН SKILL 2 (кластеризация)

Прогони Skill 2 по файлу конкурента **без** `prisms-library.md` (режим «с нуля»).

Результат сохрани как: `[название-конкурента]-perception-prisms.md`

> ⚠️ **Важно:** НЕ подсказывай модели наши кластеры. Пусть найдёт свои самостоятельно.  
> Иначе возникнет confirmation bias — модель будет искать то, что ты уже знаешь, а не то, что есть в данных.

### ШАГ 4: СРАВНЕНИЕ

Открой наш файл кластеров и файл конкурента рядом.  
Для каждого **нашего** кластера задай вопрос:

> «Есть ли у конкурента пучок с похожими драйверами?»

**Варианты ответа:**

**a) Да, почти идентичный → кластер ПОДТВЕРЖДЁН ✅**
```
Подтверждён у [конкурент]. N отзывов у них.
Драйверы совпадают: [список].
Терпимые минусы совпадают: [список].
```

**b) Похожий, но с отличиями → кластер ЧАСТИЧНО ПОДТВЕРЖДЁН ◐**
```
Частично подтверждён. Совпадает: [что].
Отличается: [что]. Возможная причина отличия: [гипотеза].
```

**c) Нет похожего → кластер ПОД ВОПРОСОМ ⚠️**
```
Не найден у [конкурент].
Возможные причины: специфика нашего продукта / артефакт данных /
недостаточно отзывов у конкурента.
```

---

Для каждого кластера **конкурента**, которого **нет у нас**:

> «Есть ли у нас хотя бы 3 отзыва с похожим пучком?»

- Если да → добавить как новый кластер ◐ с пометкой «обнаружен у конкурента [название]»
- Если нет → записать: «Кластер [название] есть у конкурента, но в наших данных не подтверждается. Не добавляем.»

### ШАГ 5: ОБНОВЛЕНИЕ prisms-library.md

Добавь данные конкурента в `prisms-library.md`. Обнови статус наших кластеров:

| Было | Стало | Условие |
|------|-------|---------|
| ● | ●● | Подтверждён у 1 конкурента |
| ●● | ●●● | Подтверждён у 2+ конкурентов |

> Кластер считается **«кластером категории»** (●●●), если подтверждён у 3+ товаров (наш + 2 конкурента).

### ШАГ 6: ПОВТОРИТЬ для следующего конкурента

Повтори шаги 1–5 для второго и третьего прямого конкурента.

---

## Работа с косвенными конкурентами (РАСШИРЕНИЕ + ИМПОРТ УГЛОВ)

Косвенные конкуренты — это другая категория товаров, но с пересекающейся аудиторией.

### ШАГ 1: ВЫБОР КОСВЕННОГО КОНКУРЕНТА

Критерии выбора:
- Другая категория, но та же аудитория
- Примеры для крема для рук: парфюмированные свечи, lip balm, travel-size косметика, hand sanitizer премиум, body mist, подарочные наборы косметики
- Достаточно отзывов (50+)

Сколько: **1–2 косвенных конкурента**.

### ШАГ 2: ПРОГОН SKILL 1 + SKILL 2

Аналогично прямым конкурентам. Режим «с нуля» (без `prisms-library.md`).

### ШАГ 3: ПОИСК НОВЫХ ПУЧКОВ

> ⚠️ **Не переносим кластеры косвенного напрямую!**

Для каждого кластера косвенного конкурента задай вопрос:

> «Есть ли в НАШИХ данных хотя бы 3 отзыва с похожим пучком?»

**Пример:** у свечей есть кластер «маленький ритуал для себя». Ищем в наших отзывах: есть ли 3+ человека, которые описывают крем как ритуал? Если да — это новый гипотетический кластер.

- Если 3+ маркеров → добавить как гипотетический кластер ◐ с пометкой «обнаружен у косвенного [название], требует проверки»
- Если < 3 маркеров → **НЕ добавлять**. Записать в заметки: «У [косвенный] есть пучок [название], в наших данных не подтверждается.»

### ШАГ 4: ИМПОРТ ФОРМУЛИРОВОК И УГЛОВ ПОДАЧИ

> ⚠️ Это НЕ для Skill 2. Это для будущего Инструмента 3.

Выпиши из отзывов косвенного конкурента:
- Какие формулировки используют покупатели?
- Какие сценарии описывают?
- Какие углы подачи работают в их карточке?

Сохрани в отдельный файл `angles-library.md`. Формат:

```
source: [название косвенного конкурента]
category: [его категория]

ФОРМУЛИРОВКИ:
- «[цитата]» — контекст: [зачем покупатель это написал]

СЦЕНАРИИ:
- [сценарий] — встречается в N отзывах

УГЛЫ ПОДАЧИ (из карточки конкурента):
- [что написано на обложке/слайдах]
- [какой приём использован]
```

### ШАГ 5: ОБНОВЛЕНИЕ prisms-library.md

Добавь данные косвенного конкурента.

> ⚠️ **НЕ меняй статус** существующих кластеров на основании косвенного конкурента.  
> Косвенный подтверждает только новые гипотезы, не валидирует старые.

## Финальный прогон товара-героя с багажом

После прогона 2–3 прямых и 1–2 косвенных конкурентов — запусти финальный прогон.

### ШАГ 1: Собери prisms-library.md

В файл должны войти данные всех прогонов: наш товар + все конкуренты.

### ШАГ 2: Прогон Skill 2 для товара-героя в режиме «с багажом»

Вход: `signal-markup.md` товара-героя + `prisms-library.md`

Модель делает всё то же самое (шаги 0–12), но дополнительно:
- Для каждого найденного кластера: проверяет, есть ли он в `prisms-library.md`
- Если есть → повышает уверенность
- Если есть новый кластер от косвенного → проверяет, подтверждается ли он нашими данными

### ШАГ 3: Результат — финальные кластеры v2

| Пометка | Значение |
|---------|----------|
| ●●● | Подтверждён у 3+ товаров (кластер категории) |
| ●● | Подтверждён у 2 товаров |
| ● | Только наши данные |
| ◐ | Гипотеза от косвенного, требует проверки |

## Порядок прогонов (рекомендуемый)

```
1. Off.Scrub (товар-герой) → кластеры v1        ← уже сделано
2. Прямой конкурент А → валидация                ← следующий шаг
3. Прямой конкурент Б → валидация + новые пучки
4. (опционально) Прямой конкурент В
5. Косвенный конкурент Г → расширение + углы
6. (опционально) Косвенный конкурент Д
7. Финальный прогон Off.Scrub с багажом → кластеры v2
```

## Частые вопросы

**Нужно ли подсказывать модели наши кластеры при прогоне по конкуренту?**  
Нет. Пусть найдёт сама. Потом сравниваем вручную. Иначе confirmation bias.

**Что если у конкурента слишком мало отзывов?**  
Меньше 50 отзывов — данные ненадёжны. Лучше найти другого конкурента. Если нет альтернативы — прогоняй, но помечай результаты как «недостаточно данных».

**Что делать, если у конкурента совсем другие кластеры?**  
Это важная информация. Означает, что наш товар действительно специфичен. Проверь: может, у нас артефакт данных? Если нет — это наше уникальное позиционирование.

**Нужно ли прогонять Skill 1 по конкурентам или можно сразу Skill 2?**  
Сначала обязательно Skill 1 — он создаёт разметку сигналов, без которой Skill 2 не работает.
