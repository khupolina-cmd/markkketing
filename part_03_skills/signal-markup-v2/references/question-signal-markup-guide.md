# Question Signal Markup Guide

Этот файл подключается только для массива вопросов.

Не читать его для массива отзывов.

Использовать вместе с:

- [core-semantic-principles.md](core-semantic-principles.md)

## Что делает question-pass

Question-pass — это облегчённая разметка.

Вопросы не должны смешиваться с отзывами, потому что:

- у них другая логика;
- в них намного больше шума;
- многие вопросы относятся к магазину, наличию, доставке или соседним товарам.

## Какие категории обычно полезны

Основные категории для вопросов:

- `🟡` ожидание
- `🟠` сомнение
- `🟣` сценарий

`🔵` сравнение допустимо только если вопрос явно сравнивает с названным объектом.

`🟢` и `🔴` обычно не нужны, потому что вопрос чаще всего не сообщает о случившемся опыте.

## Router релевантности

### Отбрасывать как не-signal

Не считать buyer-signal, если вопрос про:

- наличие;
- доставку;
- артикулы;
- наборы;
- магазин в целом;
- другой товар внутри карточки;
- служебный шум.

### Сохранять как signal

Сохранять, если вопрос касается:

- свойств товара;
- эффекта;
- состава;
- совместимости;
- сценария использования;
- ограничений использования.

## Итоговый формат question-файла

Вернуть один markdown-файл в таком порядке:

1. `Legend`
2. `Question Table`
3. `Question Stats`
4. `Trash`

### `Legend`

- `🟡` ожидание
- `🟠` сомнение
- `🟣` сценарий
- `🔵` сравнение
- `⚪` мусор

### `Question Table`

Использовать таблицу:

| id | author | date | question | signals |
|---|---|---|---|---|

Правила:

- вопрос сохраняется буквально;
- релевантные сигналы живут в колонке `signals`;
- нерелевантные и операционные вопросы не засоряют основную таблицу.

### `Question Stats`

Минимально включить:

- `Q_total`
- `Q_trash`
- `Q_clean`
- `Q_with_signals`
- `Q_signals_total`

Дальше можно дать короткий breakdown по категориям и темам.

Если тем несколько, группировать их по категориям:

- `🟡 Ожидание`
- `🟠 Сомнение`
- `🟣 Сценарий`
- `🔵 Сравнение`

### `Trash`

Собрать мусорные вопросы:

| id | author | question | reason |
|---|---|---|---|

## Нормализация времени

Если в вопросах есть даты, можно сохранить их в таблице.

Если дат нет, не выдумывать их.

Полноценную lifecycle-динамику для вопросов строить не нужно.

## Порядок исполнения

1. прочитать контекст товара;
2. применить явный фильтр времени, если он есть;
3. нормализовать вопросы в единицы;
4. отсечь нерелевантные вопросы;
5. пройти релевантные вопросы один раз;
6. разметить облегчённые категории;
7. в конце собрать `Question Stats` и `Trash`.
