# Наметки для skill по разметке сигналов

## Рекомендация по сборке

Не делать один огромный skill на всё.

Лучше собрать 2 основных skill и 1 будущий downstream skill:

1. `signal-markup`
2. `buyer-logic-interpretation`
3. `signals-to-card-strategy` позже

Так система останется проверяемой, а не превратится в чёрный ящик.

## Почему лучше 2 skill, а не 1

`signal-markup` должен мыслить как аккуратный разметчик данных.

`buyer-logic-interpretation` должен мыслить как поведенческий аналитик.

Это два разных типа работы:

- первый слой не должен фантазировать;
- второй слой как раз обязан взвешивать силу сигнала.

Если объединить их слишком рано, один слой начнёт заражать другой.

## Skill 1: signal-markup

### Задача

Превращать сырой массив отзывов, вопросов и комментариев в финальную базу сигналов.

### Когда использовать

Использовать, когда нужно:

- автоматически разметить отзывы по категориям;
- не потерять короткие, но содержательные сигналы;
- собрать базу сигналов по своему товару;
- собрать базу сигналов по конкурентам;
- подготовить основу для дальнейшей аналитики.

### Что должен уметь

1. принимать структурированный массив единиц;
2. убирать только полностью пустой мусор;
3. выделять сигналы по категориям;
4. складывать спорное в ручной разбор;
5. возвращать финальную базу сигналов;
6. считать базовую статистику;
7. поддерживать пакетный режим по нескольким товарам.

### Вход

Минимум:

- `product_id`
- `product_role`
- `source_units[]`

Желательно:

- `producer_claims`
- `source_name`
- `mode`

### Выход

- `primary_markup`
- `manual_review_queue`
- `final_signal_registry`
- `stats`

### Внутренний пайплайн

1. `normalize-input`
2. `primary-markup`
3. `manual-review`
4. `finalize-registry`
5. `category-export`

### Критичные правила

- один отзыв может содержать много сигналов;
- один фрагмент нельзя использовать дважды;
- один сигнал имеет только одну категорию;
- если категория неочевидна, не угадывать;
- не удалять сомнительные сигналы только из-за подозрения на накрутку.

## Skill 2: buyer-logic-interpretation

### Задача

Интерпретировать уже готовую базу сигналов так, как её воспринял бы обычный покупатель.

### Когда использовать

Использовать, когда нужно:

- понять, что реально отпугивает;
- увидеть, какие сигналы сильнее всего ломают желание купить;
- проверить, можно ли брать свойство в ядро подачи;
- отличить частое упоминание от реально важного фактора.

### Что должен уметь

1. принимать `final_signal_registry`;
2. сводить позитив и негатив по одному критерию;
3. учитывать ожидания и сомнения отдельно;
4. поднимать редкие, но критичные сигналы;
5. выдавать интерпретацию, а не просто пересчёт.

### Выход

- `counterweight_report`
- `critical_risks`
- `buyer_deterrence_map`
- `promise_safety_map`
- `interpretive_summary`

### Минимальные оси оценки

- `severity`
- `buyer_deterrence`
- `reversibility`
- `evidence_strength`
- `promise_risk`

## Будущий Skill 3: signals-to-card-strategy

Этот skill пока не собирать.

Он должен появиться только после стабилизации первых двух слоёв.

Его задача потом:

- брать сырые сигналы;
- брать интерпретацию;
- переводить это в решения для будущей карточки.

Сейчас это лучше держать отдельно, чтобы не испортить исследовательский контур.

## Рекомендуемый формат вызова

Пример для первого skill:

```text
Use $signal-markup

Режим: market_batch
Товар: наш крем
Роль товара: own
Источник: Wildberries
Массив: <вставка JSON или файла>
Обещания производителя: <если есть>
Нужен ручной разбор: yes
```

Пример для второго skill:

```text
Use $buyer-logic-interpretation

Товар: наш крем
Роль товара: own
База сигналов: <final_signal_registry>
Нужен противовес: yes
Нужна интерпретация глазами покупателя: yes
```

## Что стоит вынести в references

Чтобы `SKILL.md` не раздувался, лучше вынести в `references/`:

- критерии категорий;
- правила ручного разбора;
- шкалу тяжести негативов;
- правила оценки доверия к сигналу;
- примеры типичных спорных случаев;
- шаблоны итоговых отчётов.

## Что стоит держать прямо в SKILL.md

Прямо в `SKILL.md` должны остаться только:

- когда skill использовать;
- что он делает;
- что принимает на вход;
- что возвращает;
- краткий пайплайн;
- жёсткие запреты и анти-галлюцинаторные правила.

## Параллельный запуск

Это реально, но надо делать правильно.

### Что можно параллелить

1. несколько товаров одновременно;
2. несколько конкурентов одновременно;
3. категорийные модули внутри одного товара после фиксации финальной базы.

### Что нельзя параллелить без риска

1. первичную разметку одного и того же товара несколькими независимыми агентами с записью в одну базу;
2. ручной разбор одного и того же массива без версионирования;
3. интерпретацию до завершения финальной разметки.

### Практическая схема на 5 конкурентов

Нормальная схема такая:

1. поднимаются 5 отдельных jobs;
2. каждый job прогоняет свой товар до `final_signal_registry`;
3. после этого внутри каждого job можно параллелить:
   - позитив
   - негатив
   - ожидания
   - сомнения
   - сравнения
   - сценарии
4. затем по каждому товару считается противовес;
5. в самом конце строится общий рыночный синтез.

То есть параллелить 5 конкурентов одновременно вполне реально.

Главное:

не смешивать общую рыночную агрегацию с локальной разметкой одного товара.

## Узкие места, которые надо продумать до сборки skill

### 1. Идентификатор единицы

Нельзя полагаться только на имя автора.

### 2. Контракт ручного разбора

Нужно заранее решить:

- обязателен ли он всегда;
- как хранится пользовательское решение;
- как оно возвращается в финальную базу.

### 3. Шкала силы сигнала

Нужно ввести хотя бы простую шкалу:

- слабый;
- средний;
- сильный;
- критичный.

### 4. Мягкое доверие к сигналу

Лучше хранить `confidence`, чем делать бинарное "верим / не верим".

## Что должно быть в первой версии, а что нет

### В первой версии

- чёткая разметка;
- ручной разбор;
- финальная база;
- противовес;
- базовая интерпретация;
- батч по нескольким товарам.

### Не в первой версии

- генерация карточки;
- полная психомодель покупателя;
- сложная антифрод-система;
- самоподстройка весов без контроля.

## Рабочий вывод

Если собирать skill правильно, то сильнее всего выглядит не одна огромная надстройка, а связка:

`signal-markup -> buyer-logic-interpretation -> позже signals-to-card-strategy`

Это даст и скорость, и прозрачность, и масштабирование на несколько конкурентов без потери логики.
