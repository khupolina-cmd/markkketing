# Полное ТЗ для Opus Copilot - 4 запроса

## Что это за документ

Этот документ нужен как единый пакет постановки задачи для `Opus / Copilot`.

Работа должна идти не одним расплывчатым заходом, а `4 последовательными запросами` внутри одного контекста.

Логика такая:

1. сначала диагностировать и исправить контур `разметки сигналов`;
2. потом диагностировать и исправить контур `интерпретации сигналов глазами покупателя`.

Карточку товара, слайды, копирайтинг и любые решения для будущей подачи в эту задачу не включать.

## Главная цель всей работы

Нужно превратить текущие материалы в 2 полноценных рабочих skill:

1. `signal-markup`
2. `buyer-logic-interpretation`

Оба skill должны:

- работать по жёсткому контракту;
- быть пригодны для реального запуска;
- выдерживать пакетный прогон по нескольким конкурентам;
- не смешивать аналитику с проектированием карточки.

## Общий инвариант

Система должна делать вот это:

`сырой цифровой след -> чистая разметка сигналов -> аналитика -> интерпретация глазами разных типов покупателей`

Система не должна делать вот это:

`сырой цифровой след -> сразу выводы для карточки`

## Общие правила для всех 4 запросов

### Нельзя

- смешивать этап разметки и этап интерпретации;
- превращать отзыв в одну итоговую оценку;
- схлопывать разные сигналы в один общий sentiment;
- проектировать будущую карточку товара;
- выдавать красивую абстрактную архитектуру без рабочих контрактов;
- замалчивать слабые места текущей системы;
- делать вид, что один тип покупателя является универсальным.

### Нужно

- работать от уже существующих файлов и логики;
- сначала диагностировать, потом внедрять;
- сохранять максимум реального смысла;
- честно разводить спорные случаи;
- формализовать входы, выходы, статусы и пайплайн;
- заранее предусмотреть батч-режим по нескольким конкурентам.

## Запрос 1. Диагностика системы разметки сигналов

### Формулировка запроса для Opus

Проанализируй мою текущую систему разметки сигналов по отзывам и вопросам товаров.

Нужно не переписать её красиво, а провести структурную диагностику:

- что в системе уже сильное;
- что логически противоречиво;
- что недоопределено;
- какие места ломаются при автоматизации;
- какие места ломаются при масштабировании на несколько конкурентов;
- какие правила нужно ужесточить до внедрения в skill.

Работай строго в рамках слоя `разметки сигналов`.

Не переходи к проектированию карточки товара.

Не смешивай этот слой с интерпретацией покупателя.

### Что именно проверить

Проверь:

1. корректность категорий:
   - позитив
   - негатив
   - ожидание
   - сомнение
   - сравнение
   - сценарий
   - ручной разбор
   - мусор
2. полноту правил отбора сигналов;
3. риск дублей и пересечений;
4. слабые места кодов сигналов;
5. устойчивость ручного разбора;
6. проблему матчинга по имени;
7. пригодность системы для машинного контракта;
8. пригодность для батч-обработки нескольких товаров;
9. пригодность для параллельного запуска.

### Что нужно отдать на выходе

Отдай результат в 6 блоках:

1. `Что уже хорошо работает`
2. `Что сломается при автоматизации`
3. `Что сломается при масштабе 5+ конкурентов`
4. `Что нужно переписать как контракт`
5. `Что убрать или отложить`
6. `Минимальная исправленная архитектура skill`

### Важное требование

Если находишь слабое место, не ограничивайся критикой.

Для каждого слабого места предложи:

- конкретное исправление;
- почему оно нужно;
- что будет, если его не сделать.

## Запрос 2. Внедрение и сборка skill для разметки сигналов

### Формулировка запроса для Opus

Теперь на основе проведённой диагностики собери полноценный skill `signal-markup`.

Нужен не концепт, а практически пригодная структура skill:

- пайплайн
- правила батч-обработки
- правила ручного разбора
- анти-галлюцинаторные ограничения

Skill должен работать только на слое разметки и аналитической подготовки сигналов.

Он не должен интерпретировать сигналы как покупатель и не должен проектировать карточку товара.

### Обязательные свойства skill

Skill обязан:

1. принимать массив единиц с `unit_id`;
2. удалять только полностью пустой мусор;
3. извлекать много сигналов из одного отзыва;
4. не использовать один и тот же кусок текста дважды;
5. разводить сигналы по одной категории;
6. складывать спорное в `manual_review`;
7. возвращать финальную чистую базу сигналов;
8. поддерживать режим одного товара и режим нескольких товаров;
9. быть пригодным для одновременного прогона 5 конкурентов.

### Что нужно сделать

Собери:

1. финальный текст `SKILL.md` для `signal-markup`;
2. структуру папки skill;
3. перечень reference-файлов;
4. чёткий пример вызова skill;
5. пример входного JSON-контракта;
6. пример выходного JSON-контракта;
7. правила безопасного параллельного запуска по нескольким товарам.

### Обязательные режимы

Нужно предусмотреть:

- `single_product`
- `market_batch`
- `markup_only`
- `markup_plus_analytics`

### Обязательное ограничение

Запрещено добавлять в этот skill:

- будущую карточку;

- интерпретацию личности покупателя.

## Запрос 3. Диагностика системы интерпретации покупателя

### Формулировка запроса для Opus

Теперь отдельно проанализируй второй контур системы: интерпретацию сигналов глазами покупателя.

Нужно диагностировать, как превратить сухую базу сигналов в модель восприятия товара человеком.

Но здесь нельзя делать вид, что существует один универсальный покупатель.

Нужно специально спроектировать `ветвящуюся интерпретацию`:

- несколько синтетических типов покупателей;
- разные пороги чувствительности к рискам;
- разные триггеры доверия;
- разные реакции на негатив, сомнения, сценарии и обещания.

Это не должно имитировать настоящую репрезентативную выборку населения.

Это должно быть `аналитическим синтетическим тестированием`, чтобы обнаруживать скрытые факторы, которые в плоской статистике теряются.

### Что именно нужно проверить

Проверь:

1. можно ли вообще формализовать слой интерпретации без псевдонаучности;
2. какие оси восприятия реально нужны;
3. какие типы покупателей стоит моделировать;
4. как учитывать аллергию, чувствительность к риску, цену ошибки, недоверие к обещаниям, внимание к упаковке, внимание к сценарию использования;
5. как отличать слабый негатив от критичного стоп-фактора;
6. как не скатиться в примитивную формулу `3 хорошо > 2 плохо`;
7. как связать интерпретацию с сигналами, не выдумывая лишнего;
8. как сделать этот слой пригодным для отдельного skill.

### Важная рамка

Интерпретация должна опираться не только на психологику вообще, но и на:

- стереотипы потребительского поведения;
- разную чувствительность к типам риска;
- контекст покупки;
- воспринимаемую цену ошибки;
- потребительские эвристики;
- внимание к редким, но пугающим сигналам.

### Что нужно отдать на выходе

Отдай результат в 7 блоках:

1. `Можно ли это формализовать`
2. `Какие оси интерпретации обязательны`
3. `Какие ветки покупателей нужны в первой версии`
4. `Где риск псевдопсихологии и как его снизить`
5. `Как считать критичные сигналы`
6. `Минимальная рабочая архитектура interpretation-layer`
7. `Что пока нельзя обещать этой системе`

### Обязательное требование

Не выдавай "среднего покупателя" как единственный ответ.

Нужно предложить ветвящуюся модель.

Минимум - `3-6 синтетических buyer-profile`.

Примеры допустимых профилей:

- риск-чувствительный;
- чувствительный к составу и побочкам;
- прагматик цена/результат;
- эстет и сенсорный покупатель;
- подарочный сценарий;
- скептик к маркетинговым обещаниям.

Если предложишь другой набор, обоснуй его лучше.

## Запрос 4. Внедрение и сборка skill для интерпретации покупателя

### Формулировка запроса для Opus

Теперь на основе диагностики собери полноценный skill `buyer-logic-interpretation`.

Skill должен принимать уже готовую финальную базу сигналов и строить интерпретацию не как одну итоговую сводку, а как ветвящуюся систему восприятия разными синтетическими типами покупателей.

Нужна реализация уровня полноценного skill:

- `SKILL.md`
- входной контракт
- выходной контракт
- список profile-веток
- правила оценки силы сигнала
- правила противовеса
- правила чтения критичных редких сигналов
- правила батч-обработки по нескольким товарам

### Что обязан уметь этот skill

1. принимать `final_signal_registry`;
2. работать только на основе уже найденных сигналов;
3. строить не один вывод, а несколько профильных трактовок;
4. выделять сигналы, которые для одних профилей неважны, а для других стоп-фактор;
5. формировать карту:
   - что замечают первым;
   - что повышает доверие;
   - что вызывает сомнение;
   - что отпугивает;
   - что делает обещание небезопасным;
6. поддерживать прогон по нескольким конкурентам;
7. не переходить к дизайну карточки товара.

### Минимальная модель интерпретации

Нужно предусмотреть:

- `signal severity`
- `buyer deterrence`
- `reversibility`
- `promise risk`
- `evidence strength`
- `criterion axis`
- `buyer profiles`

### Требование к ветвлению

Интерпретация должна быть не плоской, а ветвящейся.

То есть один и тот же сигнал может читаться по-разному в зависимости от профиля:

- для части профилей это слабый шум;
- для части профилей это барьер;
- для части профилей это стоп-фактор.

### Что нужно собрать

Собери:

1. финальный текст `SKILL.md` для `buyer-logic-interpretation`;
2. profile-модель первой версии;
3. шкалу силы сигнала;
4. правила противовесного фильтра v2;
5. пример структуры итогового отчёта;
6. пример вызова skill;
7. правила пакетной обработки нескольких товаров и нескольких конкурентов.

### Обязательное ограничение

Этот skill не должен:

- размечать сигналы заново;
- спорить с финальной базой разметки;
- строить карточку;
- писать слайды;
- выдавать маркетинговые решения вместо интерпретации.

## Требование к батч-режиму и конкурентам

Это требование относится к запросам 2 и 4.

Нужно специально предусмотреть архитектуру, где можно:

- гонять сразу 5 конкурентов;
- гонять свой товар плюс несколько конкурентов;
- получать отдельный итог по каждому товару;
- потом строить общий рыночный слой поверх завершённых отдельных прогонов.

### Безопасная схема

Каждый товар должен идти как отдельный pipeline:

1. `normalize`
2. `primary markup`
3. `manual resolve`
4. `final registry`
5. `category analytics`
6. `buyer interpretation`

После этого можно собирать:

- `product summary`
- `competitor summary`
- `market comparison`

### Что нельзя делать параллельно

- менять одну и ту же базу сигналов несколькими потоками;
- одновременно вести ручной разбор одной и той же очереди без версий;
- начинать buyer-interpretation до фиксации final registry.

## Что особенно важно для второго контура

Это самое важное место всей постановки.

Нужно прямо донести Opus следующее:

Слой интерпретации не должен делать вид, что существует одно универсальное человеческое восприятие.

Нужно моделировать несколько синтетических типов покупателей не ради ложной репрезентативности, а ради обнаружения скрытых факторов выбора.

Пример:

один отзыв об аллергии может почти ничего не значить для нейтрального покупателя, но быть сильнейшим стоп-фактором для профиля с высокой чувствительностью к рискам состава и реакций.

То есть система должна уметь говорить не только:

`этот сигнал редкий`

но и:

`этот сигнал редкий, но для такого-то профиля он критичен`

Именно это должно стать ядром второго skill.
