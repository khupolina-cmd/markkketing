# ТЗ для Opus по системе сигналов

## Цель

Нужно собрать рабочую систему, которая автоматизирует разметку рыночных сигналов по отзывам, вопросам и другим текстовым следам под карточками товаров.

Система должна:

- быстро извлекать сигналы из массива;
- разводить их по чистым категориям;
- считать статистику по сигналам;
- выделять аномалии и встречные данные;
- отдельно интерпретировать результат так, как его воспринял бы обычный покупатель.

## Главный принцип

Система не должна схлопывать отзыв в одну оценку.

Базовая единица анализа - `сигнал`.

Один отзыв может содержать несколько сигналов разных типов.

## Что нужно получить на выходе

Нужно реализовать минимум 2 самостоятельных контура:

1. `Автоматизированная разметка сигналов`
2. `Интерпретация сигналов глазами покупателя`

Будущую карточку товара, слайды, тексты и креативные решения в эту реализацию не включать.

Это отдельная очередь работ.

## Обязательные категории сигналов

- позитив
- негатив
- ожидание
- сомнение
- сравнение
- сценарий
- ручной разбор
- мусор

## Ключевые правила системы

1. Работать от максимального сбора сигналов.
2. Удалять только полностью пустые реплики.
3. Если во фразе есть хотя бы один конкретный смысл, это сигнал.
4. Один отзыв может давать сколько угодно сигналов.
5. Один и тот же кусок текста нельзя использовать дважды.
6. Каждый сигнал может попасть только в одну категорию.
7. Если категория неочевидна, фрагмент отправляется в `ручной разбор`.
8. Статистика считается от немусорной базы, а не от числа сигналов.
9. Подозрение на накрутку не означает автоматическое удаление сигнала.

## Входной контракт

Система должна принимать на вход массив единиц в явном структурированном виде.

Минимальное поле на единицу:

```json
{
  "unit_id": "wb-123-review-045",
  "product_id": "123",
  "product_role": "own",
  "source_type": "review",
  "author_name": "Покупатель",
  "date": "2026-04-11",
  "rating": 5,
  "raw_text": "Текст отзыва...",
  "raw_html": "<div>...</div>"
}
```

### Обязательные требования к входу

- у каждой единицы должен быть `unit_id`;
- должно быть понятно, к какому товару относится единица;
- должно быть понятно, свой это товар или конкурент;
- система не должна матчить сигналы только по имени автора;
- если есть HTML, нужно сохранять и raw HTML, и извлечённый текст.

### Дополнительные входы

Отдельным блоком система должна уметь принимать:

- обещания производителя;
- название товара;
- краткие заметки по карточке;
- список конкурентов;
- режим работы: `single_product` или `market_batch`.

## Выходной контракт

Нужны как человекочитаемые, так и машинные выходы.

### Обязательные артефакты

1. `primary_markup`
2. `manual_review_queue`
3. `final_signal_registry`
4. `category_reports`
5. `counterweight_report`
6. `buyer_interpretation_report`
7. `market_summary`, если запущено несколько товаров

### Рекомендуемый машинный формат

```json
{
  "signal_id": "sig-001",
  "unit_id": "wb-123-review-045",
  "product_id": "123",
  "product_role": "own",
  "category": "негатив",
  "code": "мят.уп.",
  "theme": "мятая упаковка",
  "quoted_fragment": "пришел в мятой упаковке",
  "full_quote": "Полный текст отзыва...",
  "status": "final",
  "confidence": 0.78,
  "criterion_axis": "упаковка"
}
```

## Пайплайн

### Этап 1. Нормализация массива

Что нужно сделать:

- принять массив;
- проверить обязательные поля;
- проставить `unit_id`, если его нет;
- убрать технический мусор;
- сохранить единый массив для анализа.

### Этап 2. Первичная разметка сигналов

Что нужно сделать:

- удалить только полностью пустые реплики;
- выделить все остальные сигналы;
- разложить их по категориям;
- всё спорное отправить в `manual_review_queue`.

Обязательный выход:

- количество всех единиц;
- количество мусора;
- количество единиц в работе;
- количество извлечённых сигналов;
- очередь ручного разбора.

### Этап 3. Разрешение ручного разбора

Система должна поддерживать 2 режима:

#### `strict`

Все спорные случаи остаются на ручное подтверждение пользователя.

#### `fast`

Система может авторазруливать только низкорисковые неоднозначности, но обязана помечать такие решения как `auto_resolved`.

По умолчанию использовать `strict`.

### Этап 4. Финальная база сигналов

После ручного разбора система должна собрать чистую финальную базу:

- без мусора;
- без дублей одного и того же фрагмента;
- с одной категорией на каждый сигнал;
- с единым знаменателем `N_work_final`.

### Этап 5. Категорийные модули

Нужны отдельные модули:

- `positive-analysis`
- `negative-analysis`
- `expectation-analysis`
- `doubt-analysis`
- `comparison-analysis`
- `scenario-analysis`

Требование:

эти модули не ищут сигналы заново, а работают только по уже размеченной финальной базе.

### Этап 6. Противовесный фильтр

Нужно сводить позитив и негатив по одному критерию.

Пример критериев:

- запах
- увлажнение
- текстура
- упаковка
- стойкость
- переносимость

Минимальная логика:

- `1:1` = нейтрально или спорно
- `2:1` = слабо
- `3:1` = условно рабоче
- `4:1+` = сильная опора

Важно:

эту шкалу не считать финальной истиной, а использовать как черновую эвристику.

### Этап 7. Интерпретация глазами покупателя

Это отдельный модуль.

Он не должен просто пересчитывать частоты.

Он должен оценивать:

- что человек заметит первым;
- что реально отпугивает;
- какие сигналы ломают желание купить;
- какие сигналы лишь немного ухудшают фон;
- какие ожидания критичны;
- какие сомнения надо обязательно снимать в карточке позже.

Минимальные оси интерпретации:

- `severity`
- `buyer_deterrence`
- `reversibility`
- `evidence_strength`
- `promise_risk`

### Этап 8. Сводка по рынку

Если запускается несколько товаров, система должна отдельно собрать:

- итог по каждому товару;
- общие сигналы категории;
- различия между своим товаром и конкурентами;
- окно отличия;
- повторяющиеся барьеры покупки.

## Жёсткие ограничения

Нельзя:

- генерировать новые сигналы, которых нет в тексте;
- менять категорию сигнала без явного основания;
- схлопывать ожидания и сомнения в один мешок;
- считать весь позитив равновесным всему негативу;
- автоматически выкидывать сомнительные отзывы как фейк;
- переходить к проектированию карточки внутри этой задачи.

## Требования к качеству

Реализация считается годной, если:

1. один длинный отзыв может быть разрезан на несколько сигналов;
2. короткий, но содержательный фрагмент не теряется;
3. спорные случаи не угадываются, а честно маркируются;
4. категория не дублируется;
5. статистика считается от корректного знаменателя;
6. редкие, но критичные сигналы не тонут в массиве позитива;
7. финальный отчёт можно читать и как аналитику, и как основу для последующего стратегического этапа.

## Известные слабые места, которые нужно закрыть

### 1. Матчинг по имени

Если сейчас логика завязана на имя автора, её нужно заменить на `unit_id`.

### 2. Блок ожиданий и сомнений

Нужно чётко развести:

- что человек ожидал;
- по каким признакам он судил;
- чего он боялся до покупки;
- что оказалось фактом после покупки.

### 3. Уровень доверия к сигналу

Нужно ввести мягкий вес доверия, а не бинарное удаление.

### 4. Вес критичных негативов

Система должна различать:

- мелкий дискомфорт;
- раздражающий недостаток;
- сильный стоп-фактор;
- риск для здоровья или безопасности.

## Параллельный запуск

Параллельный запуск нужен и допустим.

### Разрешить

- прогон нескольких товаров одновременно;
- прогон нескольких конкурентов одновременно;
- прогон категорийных модулей параллельно после фиксации финальной базы по товару.

### Не разрешать

- параллельное изменение финальной базы сигналов одного и того же товара несколькими потоками;
- параллельный ручной разбор одной и той же очереди без блокировки версий.

### Рекомендуемая схема

1. каждый товар обрабатывается как отдельный job;
2. внутри job идут этапы `normalize -> primary_markup -> manual_resolve -> final_registry`;
3. после `final_registry` запускаются параллельно категорийные модули;
4. затем отдельно собирается `counterweight_report`;
5. после завершения всех jobs строится `market_summary`.

## Режимы использования

Нужно поддержать минимум 3 режима:

### `markup_only`

Только разметка и финальная база сигналов.

### `markup_plus_analytics`

Разметка плюс категорийные сводки и противовес.

### `full_market_read`

Разметка, аналитика и интерпретация по нескольким товарам.

## Что можно отложить

На вторую очередь:

- адаптивные формулы весов;
- самообучение на ручных правках;
- продвинутую детекцию накрутки;
- генерацию формулировок для карточки;
- визуальный dashboard.

## Финальная формулировка задачи для реализации

Собрать модульную систему, которая превращает сырой массив отзывов и вопросов по товару и конкурентам в:

- чистую базу сигналов;
- категорийную аналитику;
- противовесную проверку свойств;
- интерпретацию глазами покупателя;

без перехода к этапу создания будущей карточки товара.
