# Наметки для skill и параллельного прогона конкурентов

## 1. Что здесь вообще должно быть skill'ом

В твоём случае skill не должен быть “один огромный мозг на всё”.

Лучше делать skill как **оркестратор модулей**, а не как одну монолитную инструкцию.

То есть skill должен:
- понимать этап pipeline;
- знать, какой prompt-модуль вызвать;
- знать, какой артефакт должен появиться после шага;
- не смешивать extraction, counting, weighting и strategy.

---

## 2. Самая здравая форма skill

Не один супер-файл, а такая схема:

1. `SKILL.md` — назначение, правила, flow.
2. `/prompts/` — модульные prompt-файлы по этапам.
3. `/schemas/` — форматы входов/выходов.
4. `/examples/` — 1–2 мини-примера.
5. `/playbooks/` — готовые режимы запуска.

---

## 3. Какой должен быть flow skill

### Режим A. Один товар
Используется для отладки логики.

Шаги:
1. принять массив HTML / reviews / Q&A;
2. сделать первичную разметку сигналов;
3. вынести спорное в ручной разбор;
4. после ручного решения собрать финальную разметку;
5. прогнать category-модули;
6. собрать карту рисков / ценностей / сценариев / сравнений;
7. собрать buyer interpretation layer;
8. перевести в стратегический вывод для карточки.

### Режим B. Один конкурент
То же самое, но на выходе обязательно:
- `competitor_report.md`
- `competitor_signals.json`
- `competitor_strategy_risks.md`

### Режим C. Пул конкурентов
На выходе должен быть общий синтез:
- что повторяется по рынку;
- что банально;
- что тревожит покупателей;
- где окно отличия;
- что не доказывается рынком;
- какие сигналы требуют ответа в карточке.

---

## 4. Какие модули реально нужны

### Модуль 1. Primary tagging
Задача:
разложить материал на позитив / негатив / ожидание / сомнение / сравнение / сценарий / ручной разбор.

### Модуль 2. Manual resolution
Задача:
принять ручное распределение спорных случаев и собрать финальную разметку.

### Модуль 3. Category expansion
Задача:
по финальной разметке развернуть сигналы в полные цитаты и темы.

Подмодули:
- positive
- negative
- expectations
- comparisons
- scenarios
- optional: doubts as separate or merged logic

### Модуль 4. Promise validation
Задача:
сверить обещания производителя с реально подтверждаемыми сигналами.

### Модуль 5. Human weighting
Задача:
оценить, что реальный покупатель сочтёт важным, тревожным, стоп-факторным, проходным.

### Модуль 6. Strategy export
Задача:
перевести всё в:
- центральную ставку;
- вторичные опоры;
- барьеры;
- запрещённые ставки;
- принцип отличия;
- порядок раскрытия карточки.

---

## 5. Где нужен ручной шаг и не надо это прятать

Ручной шаг обязателен минимум в 4 местах:
1. распределение ручного разбора;
2. проверка спорных/опасных негативов;
3. подтверждение сильных продуктовых обещаний;
4. финальный выбор стратегического ядра.

Если это убрать и сделать вид, что всё можно автоматизировать честно, качество резко упадёт.

---

## 6. Что хранить как артефакты

На каждого конкурента лучше хранить отдельную папку:

```text
/competitors/{slug}/
  raw_input.html
  tagging_stage1.md
  manual_resolution.md
  tagging_final.md
  positive_report.md
  negative_report.md
  expectations_report.md
  comparisons_report.md
  scenarios_report.md
  promise_validation.md
  human_weighting.md
  strategy_export.md
  summary.json
```

Это даст:
- воспроизводимость;
- одинаковый формат;
- удобный merge;
- возможность прогонять не весь рынок заново.

---

## 7. Реально ли гнать 5 конкурентов одновременно

### Да, но только при правильном разрезе задачи

Параллелить лучше **не весь мозг сразу**, а **изолированные однотипные прогоны**.

Правильный параллельный юнит:
- один конкурент;
- один и тот же pipeline;
- один и тот же набор модулей;
- один и тот же формат выхода.

Неправильный параллельный юнит:
- 5 конкурентов в одном гигантском контексте;
- общий ресерч, где в одном проходе смешаны все карточки, сигналы и выводы.

Так контекст быстро пухнет и модель начинает мазать.

## 8. Где это реально делать

### В Copilot
Copilot cloud agent умеет работать в фоне, исследовать репозиторий, строить план, вносить изменения в ветку и открывать pull request. Он отдельно отличает cloud agent от agent mode в IDE, а его поведение можно усиливать custom instructions и custom agents. Но Copilot считает premium requests за чаты и отдельные agent/cloud-сессии, поэтому использовать его как массовый оркестратор на 5–10 параллельных исследовательских задач не самый экономный вариант. citeturn526509view0turn526509view1turn526509view2

### В Codex
Codex уже поддерживает параллельные агенты и отдельные worktrees в приложении, а subagent workflows позволяют запускать специализированных агентов параллельно и потом собирать их результаты. Для твоего кейса это куда ближе к идеальному режиму “пять конкурентов — пять параллельных веток обработки”. citeturn526509view4turn526509view5turn526509view3

## 9. Практический вывод по параллели

### Реалистично сейчас
- 1 товар — для отладки;
- 3–5 конкурентов — как пилотный батч;
- отдельный итоговый merge после завершения всех одиночных прогонов.

### Не стоит делать сразу
- 70–80 конкурентов одним куском;
- один общий агент на всё поле;
- общий файл без стандартизованных competitor reports.

### Лучший порядок
1. руками и со мной закрепить систему;
2. в Copilot проверить архитектуру и собрать pipeline spec;
3. в skill заложить единый формат артефактов;
4. в Codex или аналогичном агентном режиме запускать конкурентов пачками;
5. после пачки — делать market synthesis поверх нормализованных отчётов.

---

## 10. Минимальная версия skill на старт

Если делать не идеально, а умно и быстро, стартовый skill должен уметь только 4 вещи:

1. принять один competitor input;
2. прогнать его по фиксированным этапам;
3. сохранить стандартизованные артефакты;
4. не смешивать сырой слой и стратегический слой.

Этого достаточно, чтобы потом наращивать автоматизацию без переписывания всей системы.

---

## 11. Формула здравой оркестрации

Сначала:
- один товар;
- одна схема;
- один стандартизованный выход.

Потом:
- батч из 3–5 конкурентов;
- одинаковые отчёты;
- общий merge.

И только после этого:
- масштабирование;
- параллельные агенты;
- автоматизированный market synthesis.
