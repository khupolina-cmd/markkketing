# ТЗ для GitHub Copilot + Claude Opus: собрать pipeline системы сигналов рынка

## 1. Контекст задачи

У нас уже есть не нулевая логика системы.
Её не нужно придумывать с потолка.

Нужно:
- проверить архитектуру на логические дыры;
- развести слои системы;
- собрать это в воспроизводимый pipeline;
- подготовить структуру под skill / reusable prompt system / future Codex usage.

Важно:
не перепридумывай бизнес-логику без причины.
Сначала найди противоречия и слабые места в текущем контуре, потом предложи улучшения.

---

## 2. Что лежит в репозитории

Ожидаемые входные файлы:
- `SYSTEM_MARKET_SIGNALS.md` — паспорт системы;
- `SIGNAL_TAGGING_RULES.md` — правила первичной разметки;
- `PROMPT_POSITIVE.md`
- `PROMPT_NEGATIVE.md`
- `PROMPT_EXPECTATIONS_DOUBTS.md`
- `PROMPT_COMPARISONS.md`
- `PROMPT_SCENARIOS.md`
- `PROMPT_TAGS.md`
- `MARKET_RESEARCH_DECISION_LAYER.md`

Если структура отличается — сначала зафиксируй фактическую структуру, не ломай её молча.

---

## 3. Главная задача Copilot

Нужно собрать архитектуру pipeline из нескольких этапов, где каждый этап имеет:
- чёткий вход;
- чёткий выход;
- отдельную ответственность;
- понятные критерии ошибки.

Pipeline должен быть пригоден для работы:
- по одному товару;
- по одному конкуренту;
- по пачке конкурентов;
- в будущем — для параллельного прогона по нескольким конкурентам.

---

## 4. Что именно нужно сделать

### Этап 1. Архитектурный аудит
Прочитай все файлы и собери документ `ARCHITECTURE_REVIEW.md`, где есть:
1. что система делает сейчас;
2. где слои смешаны;
3. какие есть логические противоречия;
4. какие части дублируют друг друга;
5. что уже хорошо и не требует переписывания;
6. какие недостающие сущности нужны для pipeline.

### Этап 2. Сборка слоёв pipeline
Собери документ `PIPELINE_SPEC.md` со стадиями:
1. ingest;
2. primary tagging;
3. manual review resolution;
4. per-category expansion;
5. aggregation;
6. promise validation;
7. human-weight interpretation;
8. strategic export.

Для каждой стадии опиши:
- цель;
- вход;
- выход;
- запрещённые действия;
- точки сбоя;
- формат артефакта.

### Этап 3. Нормализация форматов
Собери единый документ `DATA_CONTRACTS.md`.

В нём опиши минимальные форматы:
- `raw_unit`
- `signal`
- `manual_case`
- `category_report`
- `competitor_report`
- `market_synthesis`
- `strategy_passport`

Для каждого формата укажи поля, типы и обязательность.

### Этап 4. Реестр модулей / prompts / skills
Собери документ `MODULE_REGISTRY.md`, где будет:
- какие prompt-модули уже есть;
- что делает каждый модуль;
- на каком этапе pipeline он вызывается;
- какой у него вход;
- какой у него выход;
- где сейчас есть пересечения или пробелы.

### Этап 5. Черновая структура skill
Собери документ `SKILL_BLUEPRINT.md`.

Нужен не финальный polished skill, а каркас:
- назначение;
- пошаговый flow;
- ограничения;
- expected inputs;
- expected outputs;
- как подключать модульные промпты;
- какие шаги должны оставаться ручными.

---

## 5. Принципы, которые нельзя ломать

1. Один отзыв может содержать несколько сигналов.
2. Один и тот же текстовый фрагмент нельзя класть в две категории сразу.
3. Если категория спорная — случай идёт в ручной разбор.
4. Сначала извлечение сигнала, потом интерпретация.
5. Частота сигнала и вес сигнала — разные вещи.
6. Система не ищет “истину рынка”, а моделирует вероятное чтение разумного покупателя.
7. Финальный смысл исследования — не набор наблюдений, а решение по карточке.

---

## 6. Что особенно важно проверить

Проверь и явно ответь:
1. где сейчас смешаны сырой анализ и стратегическая интерпретация;
2. нужен ли отдельный слой `human_weighting`;
3. как отделить категорию сигнала от его силы влияния на покупку;
4. как лучше хранить подозрения на накрутку: как флаг, как коэффициент доверия или как отдельный риск-слой;
5. как избежать дублирования между промптом ожиданий, сомнений, сценариев и почти-цитатных формулировок;
6. как сделать систему воспроизводимой на 70–80 конкурентах без ручного хаоса.

---

## 7. Ограничения

Не делай этого:
- не подменяй существующую систему generic sentiment analysis;
- не упрощай всё до “positive/negative score”;
- не предлагай абстрактный enterprise-архитектурный туман без реальных файлов и этапов;
- не пытайся сразу написать финальный production-grade код, если сначала не закрыт pipeline spec;
- не выдумывай данные, которых нет в файлах.

---

## 8. Желаемый формат ответа Copilot

Ответ должен быть не в чате простынёй, а в виде созданных/обновлённых файлов:
- `ARCHITECTURE_REVIEW.md`
- `PIPELINE_SPEC.md`
- `DATA_CONTRACTS.md`
- `MODULE_REGISTRY.md`
- `SKILL_BLUEPRINT.md`

Если считаешь нужным — добавь ещё:
- `OPEN_QUESTIONS.md`
- `RISKS_AND_FAILURES.md`

Но только если там реально новое содержание.

---

## 9. Критерии годности результата

Результат годен, если после него можно:
1. показать, где кончается анализ и начинается интерпретация;
2. прогнать одного конкурента от входа к выходу без смысловой каши;
3. одинаково обработать нескольких конкурентов по одной схеме;
4. собрать итоговую market synthesis без потери структуры;
5. использовать эти документы как основу для skill / Codex / дальнейшей автоматизации.

Результат не годен, если после него всё ещё непонятно:
- что является единицей данных;
- какие этапы есть в pipeline;
- какие файлы нужны;
- где ручной шаг обязателен;
- где в системе рождается решение по карточке.

---

## 10. Первый рабочий ход для Copilot

Начни не с кода.

Сначала:
1. прочитай все документы репозитория по этой системе;
2. выпиши противоречия и пробелы;
3. предложи целостную архитектуру pipeline;
4. только потом раскладывай её по файлам.

Если видишь два возможных архитектурных пути — не смешивай их.
Покажи 2 отдельных варианта с trade-offs.
